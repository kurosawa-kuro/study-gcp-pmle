PMLEは試験範囲・出題思想が更新され得るので、まず現在の公式試験ガイドを基準にして、そこから「合格直結型の教科書仕様」に落とし込みます。一般教科書ではなく、出題判断に寄せた設計に絞ります。

思考時間: 9s
結論

PMLE対策用の教科書は、**「技術を理解する本」ではなく、「試験で選択肢を切るための判断辞書」**として設計するのがよいです。

つまり特徴はこれです。

各技術を “説明” するのではなく、PMLEの問題文でその技術を選ぶ条件・選ばない条件・ひっかけ条件まで定義する教科書。

PMLEは、単なるML知識よりも、Google Cloud上で、どの制約ならどのAI/ML構成を選ぶかを問う試験です。公式ガイドでも、BigQuery ML、AutoML、Vertex AI、Model Garden、Agent Builder、Feature Store、Pipelines、Model Monitoring、Responsible AI などが、設計・運用判断として並んでいます。特に現行ガイドでは生成AI、RAG、Model Garden、Vertex AI Agent Builder、生成AI評価が明示的に入っています。

理由

一般的な教科書は、

BigQuery MLとは何か
Vertex AIとは何か
AutoMLとは何か
Feature Storeとは何か

を説明します。

しかしPMLE合格に直結する本は、そこでは足りません。

PMLEで必要なのは、

問題文に「SQL中心」「大量構造化データ」「低コード」「BigQuery内で完結」と出たら BigQuery ML を疑う。
「表形式・画像・テキストで、カスタム実装より短期構築」が強ければ AutoML。
「独自コード・分散学習・GPU/TPU・フレームワーク指定」なら Vertex AI Custom Training。
「RAG」「社内文書検索」「生成AIアプリ」「管理された検索拡張」なら Vertex AI Agent Builder。
「本番後のドリフト・スキュー・継続評価」なら Model Monitoring / Explainable AI。

という選択条件の閾値です。

公式ガイド上でも、出題比率は「MLパイプラインの自動化とオーケストレーション 22%」「モデル提供とスケーリング 20%」「プロトタイプのMLモデルへのスケーリング 18%」が重く、単なるモデル理論よりも、MLOps・サービング・運用判断が厚いです。

有力シナリオ：教科書の特徴設計
1. 各章を「技術説明」ではなく「試験判断単位」で作る

章タイトルは、サービス名よりも試験で問われる意思決定に寄せるべきです。

悪い例：

第3章 BigQuery ML
第4章 Vertex AI
第5章 AutoML

良い例：

第3章 構造化データを最短でモデル化する判断
第4章 AutoML / BigQuery ML / Custom Training の切り分け
第5章 本番サービングで online / batch / endpoint / container を選ぶ判断
第6章 drift / skew / bias / explainability が出たときの即断パターン

PMLEは「これは何ですか？」より、「この条件なら何を選ぶべきか？」が重要です。

2. 各技術に「選ぶ条件・選ばない条件・ひっかけ条件」を必ず付ける

たとえば BigQuery ML なら、こういう型にします。

項目	内容
技術の一言定義	BigQuery内のSQLでMLモデルを作成・評価・予測する機能
選ぶ条件	データがBigQueryにある、SQL中心、構造化データ、低コード、移動コストを避けたい
選ばない条件	複雑なDeep Learning、独自トレーニングループ、特殊なGPU/TPU制御が必要
ひっかけ	「MLだからVertex AI」と飛びつかせる問題。実際はBigQuery内で完結すべき
試験キーワード	SQL, BigQuery table, structured data, low-code, regression, classification, forecasting
閾値判断	“BigQuery内で完結できるか？” が第一分岐

これを全サービスで統一します。

3. 「比較表」を大量に入れる

PMLE対策では、単体説明より比較表が強いです。

特に必要なのはこのあたりです。

比較軸	必須度
BigQuery ML vs AutoML vs Vertex AI Custom Training	最重要
Vertex AI Pipelines vs Kubeflow Pipelines vs Cloud Composer	最重要
Batch prediction vs Online prediction	最重要
Model Monitoring vs Explainable AI vs Evaluation	最重要
Feature Store vs BigQuery table vs ad hoc preprocessing	高
Vertex AI Workbench vs Colab Enterprise vs Dataproc notebook	高
Dataflow vs Dataproc vs BigQuery preprocessing	高
Model Garden vs Agent Builder vs Custom RAG	高
GPU vs TPU vs CPU	中
A/B test vs canary vs shadow deployment	中

PMLEは選択肢に似たサービスが並ぶので、Aを知っているかより、AとBを切り分けられるかが合否に効きます。

4. 各章末に「PMLE判断フローチャート」を入れる

例：

モデル構築方式の判断

構造化データがBigQueryにある？
 ├─ Yes → SQL中心でよい？
 │   ├─ Yes → BigQuery ML
 │   └─ No  → Vertex AI / custom training
 └─ No → ラベル付きデータがあり、短期構築したい？
     ├─ Yes → AutoML
     └─ No  → Custom Training / foundation model / API検討

生成AI系の判断

既存文書を検索して回答したい？
 ├─ Yes → RAG
 │   ├─ マネージドで早く作る → Vertex AI Agent Builder
 │   └─ 独自制御が必要 → Vertex AI + Embeddings + Vector Search
 └─ No → 既存モデルを使う？
     ├─ Yes → Model Garden / Gemini API
     └─ No  → Fine-tuning / Custom model

このフローチャートが、普通の教科書との差別化になります。

5. 「問題文のシグナル語辞典」を作る

これはかなり効きます。

問題文の表現	疑うべき選択肢
SQL analyst / BigQuery tables / structured data	BigQuery ML
minimal code / no ML expertise / quick baseline	AutoML
custom model architecture / PyTorch / TensorFlow / GPU	Vertex AI Custom Training
real-time low latency prediction	Online prediction endpoint
nightly scoring / millions of records	Batch prediction
training-serving skew	pipeline preprocessing統一 / Feature Store / monitoring
feature reuse across teams	Feature Store
model lineage / audit / artifact tracking	Vertex ML Metadata / Experiments
drift / skew / feature attribution change	Vertex AI Model Monitoring
fairness / bias / responsible AI	Responsible AI / Explainable AI
RAG / enterprise search / documents / grounded response	Vertex AI Agent Builder / RAG
foundation model selection	Model Garden
CI/CD for model deployment	Cloud Build / Vertex AI Pipelines
orchestration / components / triggers	Vertex AI Pipelines / Kubeflow Pipelines

PMLE用参考書なら、この辞典は巻末ではなく、各章に埋め込むべきです。

6. 「正解になりやすい設計思想」を明文化する

PMLEには、Google Cloud試験らしい思想があります。

かなり重要なのは以下です。

思想	試験上の意味
マネージド優先	自前構築よりVertex AI / BigQuery / AutoMLを優先しやすい
既存データ所在地を尊重	BigQueryにあるならBigQuery ML/Dataflow/BigQuery連携を疑う
本番運用前提	prototypeだけでなく、monitoring / retraining / lineageまで考える
再現性重視	notebook単発ではなくpipeline化、metadata管理
責任あるAI	bias, fairness, explainability, privacyが出たら軽視しない
スケール要件で分岐	batch/online、CPU/GPU/TPU、endpoint/containerを切る
組織横断利用	Feature Store、Model Registry、Experimentsが出やすい

PMLE対策本は、技術説明の前にこの「試験の世界観」を置くべきです。

破綻条件

この教科書仕様が失敗するのは、以下の場合です。

1. サービス説明に寄りすぎる

BigQuery MLの使い方、Vertex AIの画面操作、AutoMLの概要説明だけだと、普通の入門書になります。PMLE合格直結度は低いです。

2. 模擬問題と本文が分離している

本文で技術説明、章末で問題、という構成は弱いです。
PMLE向けなら、本文中に常に

この条件なら正解候補
この条件なら除外
この表現ならひっかけ

を入れるべきです。

3. 生成AI領域を後付けにする

現行のPMLEでは、Model Garden、Vertex AI Agent Builder、RAG、生成AIソリューション評価、基盤モデル活用・ファインチューニングが試験ガイドに入っています。
旧来のMLだけで構成すると、現在のPMLE対策として弱くなります。

4. 実装ハンズオンに寄せすぎる

PMLEはコーディングスキルを直接評価する試験ではなく、PythonとCloud SQLの最低限の読解があればコードスニペットは解釈できる、という位置づけです。
なので、実装教材にしすぎると遠回りになります。

実務・行動への影響

仕様としては、以下の型で固めるのが一番よいです。

PMLE合格直結型教科書の章テンプレート
章タイトル：
  試験で問われる意思決定名にする

1. この章で切れる選択肢
  - BigQuery ML
  - AutoML
  - Vertex AI Custom Training
  - Dataflow
  - Feature Store など

2. 技術の最短定義
  - 3〜5行で十分

3. 試験で選ぶ条件
  - 問題文のキーワード
  - 制約条件
  - 正解になりやすい文脈

4. 試験で選ばない条件
  - 競合サービスに負ける条件
  - 自前構築すべき条件
  - 過剰設計になる条件

5. ひっかけパターン
  - よくある誤選択
  - 問題文の罠
  - 似たサービスとの混同

6. 判断フローチャート
  - 3〜7分岐で即断できる形

7. 代表問題パターン
  - 正解選択肢
  - 不正解選択肢を切る理由
  - 問題文のどの語が決定打か

8. 最後に覚える一文
  - “BigQuery内で完結する構造化MLならBigQuery ML”
  - “本番運用ならpipeline, registry, monitoring, retrainingまで考える”
最初に定めるべき教科書の特徴

最終的には、こう定義するとよいです。

本書は、Google Cloud Professional Machine Learning Engineer 試験に合格するため、AI/ML技術を網羅的に説明するのではなく、問題文の制約条件から正解サービス・構成・運用判断を選ぶための教科書である。各技術は「概要」「選ぶ条件」「選ばない条件」「競合技術との閾値」「ひっかけ」「代表問題パターン」に分解し、PMLEの出題比率が高い MLOps、サービング、パイプライン、モニタリング、生成AI/RAG、責任あるAI を重点的に扱う。

私の判断では、黒澤さんが作るならタイトルは普通の「PMLE教科書」ではなく、

PMLE 判断基準教科書
PMLE 合格のためのサービス選択辞典
Google Cloud ML設計 判断パターン集
PMLE 出題閾値ベース参考書

あたりが本質に近いです。
