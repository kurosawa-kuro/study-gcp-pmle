# 仕様と設計

> 機能仕様 + アーキテクチャ設計。プロジェクト全体像を伝える土台。
> 要件レベルの変更時に更新する。

## 共通前提

- **目的**: Google Cloud Professional Machine Learning Engineer (PMLE) に**合格直結**する教材を作る。「技術を網羅説明する本」ではなく、**問題文の制約条件から正解サービス・構成・運用判断を選ぶための「判断辞書」**を作る。
- **入力 / 出力**:
  - 入力 = 現行の公式試験ガイド（出題範囲・出題比率）、GCP 公式ドキュメント、出題傾向。
  - 出力 = PMLE「判断基準教科書」（Markdown）。比較表 / シグナル語辞典 / 判断フローチャートを核とする。
- **担当範囲**: 教材コンテンツ本体（`doc/` の仕様 + `src/` の章原稿）。模擬試験プラットフォームや採点アプリは**対象外**（§3 / `02_移行ロードマップ.md §5`）。
- **ユースケース**:
  1. 試験前に「この制約ならこのサービス」という**判断パターン**を習得する。
  2. 問題文のシグナル語（例: `BigQuery tables`, `low latency`, `RAG`）から正解候補を即断し、似たサービスを切り分ける。
  3. 似たサービスが選択肢に並ぶ問題で、A と B の**閾値**で除外判断する。
- **開発環境**: Markdown 執筆（`make fmt` / `make lint` で整形・静的検証）。
- **本番環境**: 配布形態は TODO（静的サイト / PDF / リポジトリ閲覧のいずれか。確定時に追記）。
- **言語 / フレームワーク**: 日本語 Markdown（英単語はそのまま可）。アプリフレームワークは持たない。
- **設定管理**: 非機密は `env/config.yaml`、ローカル秘密情報は `env/secret.yaml`、共有・本番秘密情報は Doppler（`doppler.yaml`）
- **CI/CD**: TODO（lint / リンク切れチェックの自動化を想定。確定時に追記）

## 1. アーキテクチャ

教材の「構造」を設計対象とする。データの流れは「公式試験ガイド → 判断単位への分解 → 章コンテンツ」。

```
公式試験ガイド（出題範囲・比率）
        │
        ▼
[1] 試験の世界観（正解になりやすい設計思想）   ← §1.1
        │
        ▼
[2] 試験判断単位への分解（章 = 意思決定）        ← §1.2
        │
        ├─ 比較表（似たサービスの閾値）          ← §2.2
        ├─ シグナル語辞典（問題文 → 疑う選択肢）  ← §2.3
        └─ 判断フローチャート（3〜7 分岐で即断）  ← §2.4
        │
        ▼
[3] 章テンプレート（全章で統一フォーマット）     ← §2.1
```

### コンセプト

- **判断辞書である**: 各技術を “説明” するのではなく、PMLE の問題文で**その技術を選ぶ条件・選ばない条件・ひっかけ条件**まで定義する。
- **「何ですか？」より「この条件なら何を選ぶ？」**: 章タイトルはサービス名ではなく、試験で問われる**意思決定**に寄せる。
  - 悪い例: 「第3章 BigQuery ML」「第4章 Vertex AI」「第5章 AutoML」
  - 良い例: 「構造化データを最短でモデル化する判断」「AutoML / BigQuery ML / Custom Training の切り分け」「本番サービングで online / batch / endpoint / container を選ぶ判断」
- **出題比率に重みを置く**: 単なるモデル理論より MLOps・サービング・運用判断を厚くする。

| 出題ドメイン | 比率 | 教材での扱い |
|---|---|---|
| ML パイプラインの自動化とオーケストレーション | 22% | 最重点 |
| モデル提供とスケーリング | 20% | 最重点 |
| プロトタイプの ML モデルへのスケーリング | 18% | 重点 |
| （上記以外のドメイン） | 残り | 標準 |

> 比率は現行ガイド時点の値。試験ガイド改訂時に必ず見直す（§4 鮮度要件）。

### 1.1 試験の世界観（正解になりやすい設計思想）

技術説明の前に、この「Google Cloud 試験らしい思想」を置く。各章の判断はこの思想に従う。

| 思想 | 試験上の意味 |
|---|---|
| マネージド優先 | 自前構築より Vertex AI / BigQuery / AutoML を優先しやすい |
| 既存データ所在地を尊重 | BigQuery にあるなら BigQuery ML / Dataflow / BigQuery 連携を疑う |
| 本番運用前提 | prototype だけでなく monitoring / retraining / lineage まで考える |
| 再現性重視 | notebook 単発ではなく pipeline 化、metadata 管理 |
| 責任ある AI | bias / fairness / explainability / privacy が出たら軽視しない |
| スケール要件で分岐 | batch / online、CPU / GPU / TPU、endpoint / container を切る |
| 組織横断利用 | Feature Store / Model Registry / Experiments が出やすい |

### 1.2 対象とする判断単位（章 = 意思決定）

章はサービス単位ではなく**意思決定単位**で立てる。最低限カバーする判断（章の詳細・順序は `02_移行ロードマップ.md` で確定）:

- 構造化データを最短でモデル化する判断（BigQuery ML / AutoML / Custom Training の切り分け）
- 本番サービングの選択（online / batch / endpoint / container）
- 運用判断（drift / skew / bias / explainability が出たときの即断）
- 生成 AI の判断（RAG / Model Garden / Agent Builder / Fine-tuning）
- パイプライン / オーケストレーション（Vertex AI Pipelines / Kubeflow / Composer）

## 2. 機能仕様

教材を構成する 4 つの「機能」= コンテンツ部品。

### 2.1 章テンプレート（全章で統一）

全章をこの 8 項目で統一する。本文と判断材料を分離せず、本文中に常に「選ぶ / 除外 / ひっかけ」を埋め込む。

1. **この章で切れる選択肢** — 例: BigQuery ML / AutoML / Vertex AI Custom Training / Dataflow / Feature Store
2. **技術の最短定義** — 3〜5 行で十分
3. **試験で選ぶ条件** — 問題文のキーワード / 制約条件 / 正解になりやすい文脈
4. **試験で選ばない条件** — 競合に負ける条件 / 自前構築すべき条件 / 過剰設計になる条件
5. **ひっかけパターン** — よくある誤選択 / 問題文の罠 / 似たサービスとの混同
6. **判断フローチャート** — 3〜7 分岐で即断できる形（§2.4）
7. **代表問題パターン** — 正解選択肢 / 不正解を切る理由 / 問題文のどの語が決定打か
8. **最後に覚える一文** — 例: 「BigQuery 内で完結する構造化 ML なら BigQuery ML」「本番運用なら pipeline / registry / monitoring / retraining まで考える」

各サービスの「選ぶ / 選ばない / ひっかけ」は次の型で統一する（例: BigQuery ML）:

| 項目 | 内容 |
|---|---|
| 技術の一言定義 | BigQuery 内の SQL で ML モデルを作成・評価・予測する機能 |
| 選ぶ条件 | データが BigQuery にある、SQL 中心、構造化データ、低コード、移動コストを避けたい |
| 選ばない条件 | 複雑な Deep Learning、独自トレーニングループ、特殊な GPU/TPU 制御が必要 |
| ひっかけ | 「ML だから Vertex AI」と飛びつかせる問題。実際は BigQuery 内で完結すべき |
| 試験キーワード | SQL, BigQuery table, structured data, low-code, regression, classification, forecasting |
| 閾値判断 | 「BigQuery 内で完結できるか？」が第一分岐 |

### 2.2 比較表

似たサービスの**閾値**を表で示す。PMLE は似たサービスが選択肢に並ぶため、「A を知っているか」より「A と B を切り分けられるか」が合否に効く。

| 比較軸 | 必須度 |
|---|---|
| BigQuery ML vs AutoML vs Vertex AI Custom Training | 最重要 |
| Vertex AI Pipelines vs Kubeflow Pipelines vs Cloud Composer | 最重要 |
| Batch prediction vs Online prediction | 最重要 |
| Model Monitoring vs Explainable AI vs Evaluation | 最重要 |
| Feature Store vs BigQuery table vs ad hoc preprocessing | 高 |
| Vertex AI Workbench vs Colab Enterprise vs Dataproc notebook | 高 |
| Dataflow vs Dataproc vs BigQuery preprocessing | 高 |
| Model Garden vs Agent Builder vs Custom RAG | 高 |
| GPU vs TPU vs CPU | 中 |
| A/B test vs canary vs shadow deployment | 中 |

### 2.3 シグナル語辞典

問題文の表現から疑うべき選択肢を引く辞典。**巻末ではなく各章に埋め込む**。

| 問題文の表現 | 疑うべき選択肢 |
|---|---|
| SQL analyst / BigQuery tables / structured data | BigQuery ML |
| minimal code / no ML expertise / quick baseline | AutoML |
| custom model architecture / PyTorch / TensorFlow / GPU | Vertex AI Custom Training |
| real-time low latency prediction | Online prediction endpoint |
| nightly scoring / millions of records | Batch prediction |
| training-serving skew | pipeline preprocessing 統一 / Feature Store / monitoring |
| feature reuse across teams | Feature Store |
| model lineage / audit / artifact tracking | Vertex ML Metadata / Experiments |
| drift / skew / feature attribution change | Vertex AI Model Monitoring |
| fairness / bias / responsible AI | Responsible AI / Explainable AI |
| RAG / enterprise search / documents / grounded response | Vertex AI Agent Builder / RAG |
| foundation model selection | Model Garden |
| CI/CD for model deployment | Cloud Build / Vertex AI Pipelines |
| orchestration / components / triggers | Vertex AI Pipelines / Kubeflow Pipelines |

### 2.4 判断フローチャート

各章末に 3〜7 分岐で即断できるフローを置く。

**モデル構築方式の判断**

```
構造化データが BigQuery にある？
 ├─ Yes → SQL 中心でよい？
 │   ├─ Yes → BigQuery ML
 │   └─ No  → Vertex AI / Custom Training
 └─ No → ラベル付きデータがあり、短期構築したい？
     ├─ Yes → AutoML
     └─ No  → Custom Training / foundation model / API 検討
```

**生成 AI 系の判断**

```
既存文書を検索して回答したい？
 ├─ Yes → RAG
 │   ├─ マネージドで早く作る → Vertex AI Agent Builder
 │   └─ 独自制御が必要       → Vertex AI + Embeddings + Vector Search
 └─ No → 既存モデルを使う？
     ├─ Yes → Model Garden / Gemini API
     └─ No  → Fine-tuning / Custom model
```

## 3. 設計上の決定事項

### 3.1 採用したアプローチと理由

- **判断辞書フォーマット（§2.1）を採用** — 一般的な「サービス説明本」は PMLE 合格直結度が低い。PMLE は「制約 → どの構成を選ぶか」を問うため、選ぶ / 選ばない / ひっかけ / 閾値を全章で明示する。
- **章 = 意思決定単位（§1.2）を採用** — サービス名章立てだと「切り分け」が学べない。
- **生成 AI を最初から組み込む** — 現行ガイドに Model Garden / Agent Builder / RAG / 生成 AI 評価 / 基盤モデル活用・Fine-tuning が含まれるため、後付けにしない。

### 3.2 採用しなかったアプローチ（破綻条件 / アンチパターン）

教材が「普通の入門書」に劣化する条件。これらは明示的に避ける。

1. **サービス説明に寄りすぎる** — 使い方 / 画面操作 / 概要説明だけだと合格直結度が落ちる。
2. **模擬問題と本文を分離する** — 「本文で説明、章末で問題」は弱い。本文中に常に「正解候補 / 除外 / ひっかけ」を入れる。
3. **生成 AI を後付けにする** — 旧来の ML だけで構成すると現行 PMLE 対策として弱い。
4. **実装ハンズオンに寄せすぎる** — PMLE はコーディングを直接評価しない（Python / SQL の最低限の読解でコードスニペットは解釈できる位置づけ）。実装教材化は遠回り。

> 「やらないこと」のスコープ確定版は `02_移行ロードマップ.md §5` を正とする。

## 4. 非機能要件

ソフトウェア的な性能・認証・可用性は本プロジェクト（教材コンテンツ）では原則該当しない。代わりに**コンテンツ品質**を非機能要件とする。

- **性能 / スケール**: 該当なし（静的コンテンツ）。配布形態が確定したらビルド時間 / サイズを追記。
- **セキュリティ / 認証**: 該当なし。秘密情報は持たない（`env/secret.yaml` はコミット禁止）。
- **可用性 / 監視**: 該当なし。配布が静的サイトになる場合のみ別途定義。
- **カバレッジ**: 出題ドメインを比率（§1 表）に沿って網羅する。比率の高いドメインを薄くしない。
- **鮮度**: 公式試験ガイド改訂に追従する。改訂時は出題範囲・比率・対象サービス（Model Garden / Agent Builder 等）を見直す。
- **一貫性**: 全章を §2.1 の 8 項目テンプレートで統一する。フォーマット逸脱を作らない。
- **検証可能性**: 各「選ぶ / 選ばない」判断は公式ドキュメントで裏取りできること。推測で書かない。
