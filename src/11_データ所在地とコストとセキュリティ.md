# 第11章 データ所在地・コスト・セキュリティの判断

> 意思決定: **データをどこに置き（所在地）、どう守り（セキュリティ）、どう安く回すか（コスト）。**
> region / データ所在地、VPC Service Controls / private endpoint、CMEK、IAM 最小権限 / Workload Identity、コスト最適化を切り分ける。全ドメインを横断する制約条件として、どの問題にも効く。

---

## §11.1 この章で切れる選択肢

データ所在地:
- **region 選択 / データ所在地（data residency）**

セキュリティ:
- **VPC Service Controls（境界）** / **private endpoint**
- **CMEK（顧客管理鍵）** / Google 管理鍵
- **IAM 最小権限** / **Workload Identity Federation（WIF）** vs サービスアカウント鍵
- **Sensitive Data Protection（旧 DLP）**（機密の匿名化。第8章と連携）

コスト:
- **batch vs online**、**scale-to-zero / autoscaling**、**committed use**、**モデルサイズ/蒸留**、**Vector Search 等の課金構造**

---

## §11.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| データ所在地 | データを特定の region / 国に留める要件。規制対応で選ぶ。 |
| VPC Service Controls | サービス境界を作り、境界外へのデータ持ち出しを防ぐ（exfiltration 対策）。 |
| private endpoint | エンドポイント/サービスへのアクセスを VPC 内に閉じ、公開しない。 |
| CMEK | 暗号鍵を顧客が Cloud KMS で管理する。鍵のライフサイクルを統制したいとき。 |
| IAM 最小権限 | 必要最小のロールだけ付与する原則。 |
| Workload Identity Federation | 外部 ID で GCP にアクセスさせ、**長期のサービスアカウント鍵を持たない**。 |

> ⚠ 要公式確認: 機密保護サービスは旧 Cloud DLP → Sensitive Data Protection に改称。CMEK 対応範囲・WIF の推奨や VPC-SC 対応サービスは変化し得る。現行仕様は公式ドキュメントで確認すること。

---

## §11.3 試験で選ぶ条件

### データ所在地

| 選択 | 選ぶ条件 | 試験キーワード |
|---|---|---|
| region 固定 / データ所在地 | 規制で国/地域にデータを留める / GDPR 等 | `data residency`, `region`, `regulatory`, `must stay in` |

### セキュリティ

| 選択 | 選ぶ条件 |
|---|---|
| VPC Service Controls | データ持ち出し（exfiltration）を防ぐ境界が要件 |
| private endpoint | 推論/サービスを公開せず VPC 内限定 |
| CMEK | 暗号鍵を自社で管理・統制したい |
| IAM 最小権限 | 過剰権限を避ける / 監査 |
| Workload Identity Federation | **SA 鍵を配らない** / 外部 CI や別クラウドからの安全な接続 |
| Sensitive Data Protection | PII/機密の検出・匿名化（第8章） |

### コスト最適化

| 選択 | 選ぶ条件 |
|---|---|
| batch prediction | 即時不要・大量 → online 常駐より安い（第3章） |
| scale-to-zero / autoscaling | トラフィック変動・アイドル時のコスト削減 |
| committed use / 予約 | 定常的な大量利用で割引 |
| 小型モデル / 蒸留 | 推論コスト・レイテンシ削減（第10章） |
| 課金構造の理解 | Vector Search 常駐コスト、GPU 常駐、ストレージ等を要件に合わせる |

---

## §11.4 試験で選ばない条件

- **SA 鍵の長期配布を選ばない**: 「外部からの接続」「鍵管理が負担」なら WIF。長期 SA 鍵はセキュリティ上の最終手段。
- **public endpoint を選ばない**: 「社内限定」「公開不可」なら private endpoint + VPC-SC。
- **過剰権限を選ばない**: `Owner`/`Editor` を広く付ける選択肢は IAM 最小権限に反する。
- **コストを無視した常駐を選ばない**: アイドルが多いのに常時大規模常駐（scale-to-zero / batch を検討）。

---

## §11.5 ひっかけパターン

1. **SA 鍵 vs WIF** — 「鍵の管理・漏洩が心配」「外部 CI/別クラウドから」なら WIF。SA 鍵を作って配る選択肢はひっかけ。
2. **所在地の見落とし** — 「データは国内に留める必要」があるのに multi-region やグローバルサービスを選ぶ。→ region/データ所在地を満たす構成。
3. **CMEK と VPC-SC の取り違え** — 「鍵を自社管理」は CMEK、「データ持ち出しを防ぐ境界」は VPC-SC。要件の動詞で切る。
4. **コスト最適化の取り違え** — 即時不要の大量推論に online 常駐（→ batch）。レイテンシ厳しくないアイドル多めに常時多 replica（→ scale-to-zero/autoscaling）。

---

## §11.6 判断フローチャート

**セキュリティの判断**

```
要件は？
 ├─ データを特定 region/国に留める → region 固定 / データ所在地
 ├─ データ持ち出し（exfiltration）を防ぐ → VPC Service Controls
 ├─ 公開せず VPC 内限定 → private endpoint
 ├─ 暗号鍵を自社管理 → CMEK（Cloud KMS）
 ├─ 外部/別クラウドから安全接続・鍵を配らない → Workload Identity Federation
 ├─ 過剰権限を避ける → IAM 最小権限
 └─ PII/機密の匿名化 → Sensitive Data Protection（第8章）
```

**コスト最適化の判断**

```
コストを下げたい対象は？
 ├─ 即時不要・大量推論 → batch prediction（第3章）
 ├─ アイドルが多い/変動が大きい → scale-to-zero / autoscaling
 ├─ 定常的な大量利用 → committed use / 予約割引
 └─ 推論が重い → 小型モデル / 蒸留（第10章）
```

## §11.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> 外部の CI/CD（GitHub Actions）から Vertex AI にデプロイしたい。**サービスアカウント鍵をリポジトリに置きたくない**。最適な方法は？
>
> A. SA 鍵を Secret に保存して使う
> B. **Workload Identity Federation で外部 ID に GCP アクセスを許可（鍵レス）**
> C. Owner ロールのユーザーで実行
> D. VM に SA 鍵を焼き込む

- **正解: B**。決定打は「**外部 CI から**」「**鍵を置きたくない**」= WIF（鍵レス）。
- A / D を切る: 長期 SA 鍵の配布・保管はリスク。
- C を切る: Owner は最小権限に反する。

### 問題 2

> 金融規制により、**学習データと予測は国内 region から出してはならない**。さらに境界外へのデータ持ち出しを技術的に防ぎたい。最適な構成は？
>
> A. multi-region のグローバルサービスを使う
> B. **国内 region に固定 + VPC Service Controls で境界を作り exfiltration を防ぐ**
> C. CMEK だけ設定
> D. public endpoint で IP 制限

- **正解: B**。決定打は「**国内 region から出さない**」「**持ち出しを防ぐ**」= データ所在地 + VPC-SC。
- A を切る: グローバル/multi-region は所在地要件に反する。
- C を切る: CMEK は鍵管理であって所在地・境界の解ではない。
- D を切る: IP 制限だけでは exfiltration 境界として弱い。

### 問題 3

> あるバッチ推論ジョブは**毎晩 1 回**しか動かないのに、常時 online エンドポイントを 10 replica 立てていてコストが高い。最適な是正は？
>
> A. replica を 20 に増やす
> B. **batch prediction に切替（常駐をやめる）/ または scale-to-zero 構成**
> C. CMEK を有効化
> D. より大きい GPU

- **正解: B**。決定打は「**毎晩 1 回**」「**常時 online でコスト高**」= batch 化 / scale-to-zero（第3章）。
- A / D を切る: コスト増。
- C を切る: 暗号鍵管理はコストの解ではない。

---

## §11.8 最後に覚える一文

> **所在地は region 固定、持ち出し防止は VPC-SC、鍵自社管理は CMEK、鍵レス接続は WIF、権限は最小。コストは「即時不要は batch、アイドルは scale-to-zero、重い推論は蒸留」。**
