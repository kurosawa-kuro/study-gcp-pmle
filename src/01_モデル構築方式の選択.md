# 第1章 モデル構築方式の選択

> 意思決定: **構造化データやよくあるタスクを、どの方式でモデル化するか。**
> この章は PMLE で最も頻出する「最初の分岐」を扱う。多くの問題は、まずここで方式を切れるかどうかで決まる。

---

## §1.1 この章で切れる選択肢

- **BigQuery ML（BQML）** — SQL でモデルを作る
- **AutoML（Vertex AI）** — 低コードで高品質モデルを作る
- **Vertex AI Custom Training** — 独自コードで学習する
- （対比のために）**事前学習済み API / 基盤モデル** — そもそも学習しない選択肢

> 第一分岐は「**そもそも自分で学習する必要があるか**」。既存 API（Vision / Speech / Translation / Gemini 等）で要件が満たせるなら、学習せずにそれを使うのが正解になりやすい（思想①マネージド優先）。

---

## §1.2 技術の最短定義

| 方式 | 一言定義 |
|---|---|
| BigQuery ML | BigQuery 内の SQL（`CREATE MODEL`）でモデルを作成・評価・予測する。データを動かさずに ML できる。 |
| AutoML | データとラベルを与えるだけで、Google がアーキテクチャ探索・学習・チューニングを自動で行う。表形式・画像・テキスト等に対応。 |
| Custom Training | 自分の学習コード（TensorFlow / PyTorch / scikit-learn 等）をコンテナで Vertex AI 上で実行する。アーキテクチャ・学習ループ・アクセラレータを完全制御。 |

> ⚠ 要公式確認: AutoML が対応するデータ種別（表形式 / 画像 / テキスト / 動画）と、一部が Gemini ベースの新ソリューションへ移行している点は変化が速い。現行の対応マトリクスは公式ドキュメントで確認すること。

---

## §1.3 試験で選ぶ条件

### BigQuery ML を選ぶ

| 項目 | 内容 |
|---|---|
| 技術の一言定義 | BigQuery 内の SQL で ML モデルを作成・評価・予測する |
| 選ぶ条件 | データが既に BigQuery にある / SQL 中心のチーム / 構造化データ / 低コード / データ移動を避けたい |
| 選ばない条件 | 複雑な Deep Learning / 独自トレーニングループ / 特殊な GPU・TPU 制御が必要 |
| ひっかけ | 「ML だから Vertex AI」と飛びつかせる。実際は BQ 内で完結すべき |
| 試験キーワード | `SQL`, `BigQuery table`, `structured data`, `low-code`, `SQL analyst`, `regression / classification / forecasting` |
| 閾値判断 | **「BigQuery 内で完結できるか？」が第一分岐** |

### AutoML を選ぶ

| 項目 | 内容 |
|---|---|
| 技術の一言定義 | データ + ラベルから自動でモデルを構築する低コード手段 |
| 選ぶ条件 | ML 専門家がいない / 短期間で高品質ベースラインが欲しい / 画像・テキスト・表形式 / コードを書きたくない |
| 選ばない条件 | アーキテクチャを制御したい / 既存の学習コード資産がある / SQL とデータ所在地的に BQML で足りる |
| ひっかけ | 「カスタムが高機能だから Custom Training」と誘う。要件が低コードなら AutoML が正解 |
| 試験キーワード | `minimal code`, `no ML expertise`, `quick / high-quality baseline`, `image / text classification` |
| 閾値判断 | **「コードを書かずに高品質が要るか？」** |

### Custom Training を選ぶ

| 項目 | 内容 |
|---|---|
| 技術の一言定義 | 独自コードをコンテナで Vertex AI 上で学習させる |
| 選ぶ条件 | 独自アーキテクチャ / フレームワーク指定（PyTorch / TF）/ カスタム損失・学習ループ / GPU・TPU 制御 / 分散学習 |
| 選ばない条件 | 標準的なタスクで低コードで十分 / データが BQ にあり SQL で足りる |
| ひっかけ | 「柔軟だから常に Custom」。運用負荷が上がるため、要件が低コードなら過剰設計 |
| 試験キーワード | `custom model architecture`, `PyTorch / TensorFlow`, `custom training loop`, `GPU / TPU`, `distributed training` |
| 閾値判断 | **「独自コード／フレームワーク制御が要るか？」** |

---

## §1.4 試験で選ばない条件（切り分けの要点）

- **BQML を選ばない**: Deep Learning の独自アーキ, 画像・自由テキストの高度処理, GPU/TPU 制御が要るとき。
- **AutoML を選ばない**: 学習コード資産があり再利用したい / アーキテクチャを触りたい / BQ 内 SQL で十分なとき（AutoML は過剰）。
- **Custom Training を選ばない**: 標準タスク + 低コード要件のとき（運用負荷の過剰設計）。
- **そもそも学習しない**: 汎用タスク（OCR, 翻訳, 一般物体検出, 要約・チャット）は事前学習済み API / 基盤モデルで足りることが多い。

---

## §1.5 ひっかけパターン

1. **「ML = Vertex AI」短絡** — データが BQ にあり SQL で済む回帰/分類なのに Vertex AI Custom を選ばせる。→ データ所在地（思想②）で BQML が正解。
2. **「高機能 = Custom Training」短絡** — 「最小限のコードで」「ML 専門家がいない」とあるのに Custom を選ばせる。→ 要件は低コードなので AutoML。
3. **「自前で作る」誘導** — 汎用タスク（翻訳・OCR・一般的な画像分類・チャット）なのに学習させる。→ 事前学習済み API / 基盤モデルで十分。
4. **データ移動の罠** — BQ のデータをわざわざ GCS にエクスポートして Custom 学習させる選択肢。→ 不要なデータ移動はコスト・skew リスク。BQML か BQ 直結を選ぶ。

---

## §1.6 判断フローチャート

```
そのタスクは汎用タスク（翻訳/OCR/一般画像/要約・チャット）？
 ├─ Yes → 事前学習済み API / 基盤モデル（学習しない）
 └─ No → データは構造化で、既に BigQuery にある？
     ├─ Yes → SQL 中心でよい / 標準的な回帰・分類・予測？
     │   ├─ Yes → BigQuery ML
     │   └─ No（独自アーキ・DL が必要） → Custom Training
     └─ No → コードを書かずに高品質モデルが欲しい？
         ├─ Yes → AutoML
         └─ No（独自コード・フレームワーク・GPU/TPU 制御） → Custom Training
```

## §1.7 代表問題パターン

> いずれも本書の自作シナリオ（実際の試験問題ではない）。「決定打の語」を意識して読む。

### 問題 1

> あるアナリストチームは、過去の取引データ（数 TB）をすべて BigQuery に保持している。チームは SQL に習熟しているが ML の専門家はいない。四半期売上の**予測**モデルを、できるだけ追加インフラなしで作りたい。最適な方法は？
>
> A. データを GCS にエクスポートし Vertex AI Custom Training で学習する
> B. **BigQuery ML で `CREATE MODEL` を使い時系列予測モデルを作る**
> C. Dataproc に Spark MLlib クラスタを立てる
> D. AutoML 画像分類を使う

- **正解: B**。決定打は「**BigQuery に保持**」「**SQL に習熟**」「**追加インフラなし**」（思想②データ所在地 + 低コード）。
- A を切る: 不要なデータ移動 + 運用負荷の過剰設計。
- C を切る: Spark クラスタ管理はマネージド優先に反する。
- D を切る: タスクは予測（時系列回帰）で画像分類は無関係。

### 問題 2

> スタートアップが商品写真を**カテゴリ分類**したい。社内に ML エンジニアはおらず、数週間で精度の高いモデルを本番化したい。学習データはラベル付き画像が数万枚。最適な方法は？
>
> A. **AutoML（画像分類）でモデルを学習しエンドポイントにデプロイ**
> B. ResNet を PyTorch で実装し Custom Training する
> C. BigQuery ML でロジスティック回帰
> D. 事前学習済みの汎用 Vision API をそのまま使う

- **正解: A**。決定打は「**ML エンジニアがいない**」「**数週間で**」「**ラベル付き画像**」（低コード + 短期 + 画像）。
- B を切る: 専門家不在・短期要件に対し過剰。
- C を切る: BQML は画像分類向きでなく、構造化データ前提。
- D を切る: **自社固有のカテゴリ**が必要なので汎用 API では分類体系が合わない（ここが罠）。

### 問題 3

> 研究チームが独自の Transformer アーキテクチャを論文どおりに実装し、複数 GPU で分散学習したい。フレームワークは PyTorch 指定。最適な方法は？
>
> A. AutoML 表形式
> B. BigQuery ML の DNN
> C. **Vertex AI Custom Training（PyTorch コンテナ + GPU 分散）**
> D. 事前学習済み API

- **正解: C**。決定打は「**独自アーキテクチャ**」「**PyTorch 指定**」「**複数 GPU 分散**」（思想⑥スケール分岐 + フレームワーク制御）。
- A / B を切る: 独自アーキ・フレームワーク制御ができない。
- D を切る: 新規アーキの学習が目的なので既製 API は不適。

---

## §1.8 最後に覚える一文

> **構造化データが BigQuery にあり SQL で足りるなら BigQuery ML。低コードで高品質なら AutoML。独自コード・フレームワーク・GPU/TPU 制御なら Custom Training。汎用タスクはそもそも学習せず API。**
