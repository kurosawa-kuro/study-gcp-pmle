# 第2章 カスタム学習を本番にスケールする

> 意思決定: **ノートブックのプロトタイプを、どう本番品質の学習にスケールするか。**
> 開発環境（どこで試作するか）、学習ジョブ（どう本番学習するか）、アクセラレータ（CPU/GPU/TPU）、分散戦略、ハイパーパラメータ最適化を切り分ける。出題ドメイン「プロトタイプのスケーリング（想定 18%）」の中心。

---

## §2.1 この章で切れる選択肢

開発環境:
- **Vertex AI Workbench**（マネージドノートブック）
- **Colab Enterprise**（マネージド・共有ノートブック）
- **Dataproc 上のノートブック**（Spark 前提）

学習の実行:
- **Custom Training ジョブ**（事前ビルド済みコンテナ / カスタムコンテナ）
- **分散学習**（データ並列 / モデル並列、Reduction Server）
- **Hyperparameter tuning ジョブ**（Vertex AI Vizier ベース）

アクセラレータ:
- **CPU / GPU / TPU** の選択

---

## §2.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Vertex AI Workbench | マネージドな JupyterLab 環境。BQ / GCS / Vertex と統合され、個人〜チームの試作に使う。 |
| Colab Enterprise | 企業向けマネージド Colab。共有・IAM・ランタイム管理ができ、軽量な共同試作に向く。 |
| Dataproc ノートブック | Spark / Hadoop クラスタ上のノートブック。**大規模分散データ処理**が主目的のときに使う。 |
| Custom Training ジョブ | 学習コードをコンテナで実行する Vertex AI のマネージドジョブ。試作を本番学習に昇格させる手段。 |
| 事前ビルド済みコンテナ | TF / PyTorch / sklearn 等の Google 提供コンテナ。コードだけ載せれば動く。 |
| カスタムコンテナ | 自分で Dockerfile を書く。依存が特殊・OS パッケージが要るときに使う。 |
| Reduction Server | 複数 GPU データ並列学習の all-reduce を高速化する Vertex AI の機能。 |
| Hyperparameter tuning | 探索空間を指定すると Vizier が最適なハイパーパラメータを探索する。 |

> ⚠ 要公式確認: ノートブック製品はリブランド・統合が進行中（旧 "Notebooks"、Workbench の managed/user-managed 区分、Colab Enterprise への寄せ）。現行の製品名・推奨は公式ドキュメントで確認すること。

---

## §2.3 試験で選ぶ条件

### 開発環境（どこで試作するか）

| 環境 | 選ぶ条件 | 試験キーワード |
|---|---|---|
| Vertex AI Workbench | マネージドな個人/チームの ML 試作、Vertex/BQ 統合、長時間ジョブやスケジュール実行 | `managed notebook`, `Jupyter`, `integrated with Vertex` |
| Colab Enterprise | 軽量・共有・素早い共同編集、IAM 管理されたノートブック | `shared notebook`, `collaboration`, `enterprise Colab` |
| Dataproc ノートブック | **Spark で大規模分散前処理/学習**が必要 | `Spark`, `Hadoop`, `existing Spark jobs`, `large-scale distributed` |

### アクセラレータ（CPU/GPU/TPU）

| 選択 | 選ぶ条件 | 閾値 |
|---|---|---|
| CPU | 小規模 / 表形式 / 古典的 ML / コスト最優先 | 行列演算が支配的でない |
| GPU | Deep Learning 全般、中規模、PyTorch/TF のカスタムモデル | 汎用的な DL ならまず GPU |
| TPU | **大規模な TensorFlow/JAX**、行列演算が支配的、超大規模学習 | TF/JAX + 大規模 + 高スループット |

> 決定打: 「TensorFlow / JAX」かつ「大規模」なら TPU が候補。フレームワークが PyTorch 中心、または小〜中規模なら GPU。表形式・古典 ML なら CPU で足りることが多い。

### 学習の実行・分散

| 選択 | 選ぶ条件 |
|---|---|
| 事前ビルド済みコンテナ | 標準フレームワークで依存が普通 |
| カスタムコンテナ | 特殊依存 / OS パッケージ / 独自ランタイム |
| 単一ノード | データ・モデルが 1 台に収まる |
| データ並列（マルチ GPU/ワーカー） | データが大きく学習が遅い。Reduction Server で all-reduce を高速化 |
| モデル並列 | モデルが 1 デバイスに載らないほど巨大 |
| Hyperparameter tuning | 精度を体系的に詰めたい / 手動チューニングを避けたい |

---

## §2.4 試験で選ばない条件

- **Dataproc を選ばない**: Spark が要らないのに「大規模だから Dataproc」と選ぶ。Vertex Custom Training + GPU/TPU で足りるなら過剰。
- **TPU を選ばない**: PyTorch 中心 / 小規模 / 表形式なのに TPU。コスト・適合性ともに不適（TPU は TF/JAX・大規模が前提）。
- **カスタムコンテナを選ばない**: 標準フレームワークで足りるのに毎回 Dockerfile を書く（過剰な運用負荷）。
- **手動チューニングに固執しない**: 「再現性・体系性」が問われたら Hyperparameter tuning（Vizier）を選ぶ。

---

## §2.5 ひっかけパターン

1. **「大規模 = Dataproc」短絡** — 大規模でも Spark が不要なら Vertex Custom Training（分散 + アクセラレータ）が正解。Dataproc は Spark/Hadoop 資産があるときの選択肢。
2. **「DL = TPU」短絡** — DL でも PyTorch 中心や中小規模なら GPU。TPU は TF/JAX・大規模・行列演算支配が条件。
3. **ノートブックで本番学習** — 「再現性」「スケジュール」「本番」が出たら、ノートブック単発ではなく Custom Training ジョブ / Pipeline に昇格させる（思想④再現性）。
4. **垂直スケール固執** — マシンを大きくし続ける選択肢。データ並列（水平分散）+ Reduction Server が正解のことが多い。

---

## §2.6 判断フローチャート

**開発環境の判断**

```
Spark / Hadoop の分散処理が前提？
 ├─ Yes → Dataproc 上のノートブック
 └─ No → 軽量・共有・共同編集が主目的？
     ├─ Yes → Colab Enterprise
     └─ No（本格的な ML 試作・Vertex 統合） → Vertex AI Workbench
```

**学習スケールの判断**

```
試作は終わり、本番品質で学習したい？
 └─ Yes → Custom Training ジョブ化（ノートブックから昇格）
     │
     ├─ 学習が遅い／データが大きい？
     │   ├─ Yes → データ並列（マルチワーカー/GPU）+ Reduction Server
     │   └─ モデルが1デバイスに載らない → モデル並列
     │
     ├─ フレームワーク／規模でアクセラレータを選ぶ
     │   ├─ TF/JAX かつ大規模 → TPU
     │   ├─ DL 一般（PyTorch 含む）→ GPU
     │   └─ 表形式/古典ML/小規模 → CPU
     │
     └─ 精度を体系的に詰めたい → Hyperparameter tuning（Vizier）
```

## §2.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> あるチームは PyTorch で書いた画像モデルを 1 枚の GPU で学習しているが、データ量の増加で 1 エポックに数日かかる。コードは変えたくないが学習時間を短縮したい。最適な方法は？
>
> A. マシンタイプをより大きい単一 CPU インスタンスに変更
> B. **Vertex AI Custom Training でマルチ GPU のデータ並列学習にし、Reduction Server を有効化**
> C. Dataproc に移行し Spark MLlib で再実装
> D. TPU に切り替える

- **正解: B**。決定打は「**PyTorch**」「**学習時間短縮**」「**コードは変えたくない**」。データ並列 + Reduction Server で all-reduce を高速化（思想⑥スケール分岐）。
- A を切る: CPU はむしろ遅い。垂直スケールの罠。
- C を切る: Spark 再実装はコード変更が大きく、画像 DL に不適。
- D を切る: TPU は TF/JAX 前提。PyTorch では適合性が低い（ここが罠）。

### 問題 2

> 大規模な TensorFlow モデルを、できるだけ高いスループットで学習したい。行列演算が支配的で、予算より速度を優先する。最適なアクセラレータは？
>
> A. 多数の CPU
> B. 単一 GPU
> C. **TPU**
> D. Dataproc

- **正解: C**。決定打は「**TensorFlow**」「**大規模**」「**行列演算が支配的**」「**スループット優先**」。TPU の適合条件がすべて揃う。
- A / B を切る: 規模・スループット要件に届かない。
- D を切る: Dataproc は学習アクセラレータの選択肢ではない（Spark クラスタ）。

### 問題 3

> あるモデルの精度が頭打ちで、学習率・バッチサイズ・層数の組合せを体系的に探索したい。手作業の試行錯誤は避けたい。最適な方法は？
>
> A. ノートブックで手動グリッドサーチ
> B. **Vertex AI の Hyperparameter tuning ジョブ（Vizier）で探索空間を定義**
> C. AutoML に切り替える
> D. より大きい GPU を使う

- **正解: B**。決定打は「**組合せを体系的に探索**」「**手作業を避けたい**」。Vizier によるベイズ最適化が適合（思想④再現性）。
- A を切る: 手動・再現性が低い。
- C を切る: 既存のカスタムモデルを捨てることになり、要件は「このモデルの最適化」。
- D を切る: ハードを大きくしても探索の問題は解けない。

---

## §2.8 最後に覚える一文

> **試作は Workbench、本番学習は Custom Training ジョブに昇格。速くするならまず水平分散（データ並列 + Reduction Server）。TF/JAX 大規模は TPU、それ以外の DL は GPU。チューニングは Vizier に任せる。**
