# 第5章 データと特徴量の管理

> 意思決定: **特徴量をどこで前処理し、どう共有・再利用し、training-serving skew をどう防ぐか。**
> 前処理エンジン（Dataflow / Dataproc / BigQuery）、特徴量基盤（Feature Store vs BQ テーブル vs その場処理）、skew 防止の置き場所を切り分ける。出題ドメイン「データとモデルの管理（チーム横断）」の中心。

---

## §5.1 この章で切れる選択肢

前処理エンジン:
- **Dataflow**（Apache Beam、ストリーミング + バッチ統一、サーバーレス）
- **Dataproc**（Spark / Hadoop、既存資産の移行）
- **BigQuery 前処理**（SQL で完結）
- **TensorFlow Transform（TFT）**（学習と推論で同じ変換を保証）

特徴量基盤:
- **Vertex AI Feature Store** / **BigQuery テーブル** / **その場（ad hoc）前処理**

横断テーマ:
- **training-serving skew** をどこで防ぐか

---

## §5.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Dataflow | Apache Beam ベースのサーバーレス処理基盤。ストリーミングとバッチを同じコードで書ける。 |
| Dataproc | マネージドな Spark / Hadoop クラスタ。既存の Spark/Hadoop ジョブを動かすのに向く。 |
| BigQuery 前処理 | データが BQ にあるとき、SQL（や BQML の `TRANSFORM`）で前処理を完結させる。 |
| TensorFlow Transform | 前処理ロジックをグラフに焼き込み、学習時と推論時で**同じ変換**を適用して skew を防ぐ。 |
| Vertex AI Feature Store | 特徴量を一元管理し、学習用の取得とオンライン低レイテンシ取得を両立。チーム横断で再利用。 |
| training-serving skew | 学習時と推論時で特徴量の作り方が違うために本番精度が落ちる現象。 |

> ⚠ 要公式確認: Feature Store は世代（Legacy 版と、BigQuery を基盤とする新 Feature Group / Feature View / Feature Online Store 版）で名称・推奨が変化している。現行の構成と推奨は公式ドキュメントで確認すること。

---

## §5.3 試験で選ぶ条件

### 前処理エンジン

| エンジン | 選ぶ条件 | 試験キーワード | 閾値 |
|---|---|---|---|
| Dataflow | ストリーミング / バッチ統一 / サーバーレス / Beam | `streaming`, `Apache Beam`, `real-time ingestion`, `unified batch+stream` | ストリーミングや統一処理が要るか |
| Dataproc | 既存 Spark/Hadoop 資産 / 移行 / Spark MLlib | `existing Spark jobs`, `Hadoop`, `migrate`, `Spark` | Spark/Hadoop 資産があるか |
| BigQuery 前処理 | データが BQ にある / SQL で足りる | `data in BigQuery`, `SQL`, `transform in BQ` | BQ 内で完結できるか |
| TFT | 学習・推論で変換を完全一致させたい / skew を構造的に防ぐ | `training-serving skew`, `same transformation`, `TFX` | skew を構造で防ぎたいか |

### 特徴量基盤

| 基盤 | 選ぶ条件 |
|---|---|
| Feature Store | **複数チームで特徴量を再利用** / オンライン低レイテンシ取得 / skew 防止 / 特徴量の一元管理 |
| BigQuery テーブル | 学習用の特徴量を保管・共有するが、オンライン取得や横断再利用の要件は薄い |
| その場前処理 | 単発の試作。本番・共有・再現性が要らないとき（基本は避ける） |

---

## §5.4 試験で選ばない条件

- **Dataproc を選ばない**: Spark/Hadoop 資産がないのに「大規模だから Spark」。データが BQ にあるなら BQ 前処理、ストリーミングなら Dataflow。
- **Dataflow を選ばない**: SQL で BQ 内完結できる単純なバッチ変換にわざわざ Beam パイプライン（過剰設計）。
- **Feature Store を選ばない**: 単一チーム・オフライン学習だけで再利用要件がないのに導入（運用負荷の過剰）。ただし**オンライン取得や skew**が出たら正解になりやすい。
- **その場前処理を選ばない**: 「本番」「再現性」「チーム共有」が出たら ad hoc は誤り。

---

## §5.5 ひっかけパターン

1. **「大規模 = Dataproc」短絡** — 大規模でも Spark 資産がなければ Dataflow か BQ。Dataproc は既存 Spark/Hadoop の移行が決定打。
2. **skew の置き場所** — クライアントや学習スクリプトだけで前処理する選択肢。→ 学習と推論で同じ変換を保証する場所（TFT / Feature Store / サービングコンテナ内前処理）が正解（思想③本番運用前提）。
3. **Feature Store の過剰/過小** — 「複数チームで再利用」「オンライン低レイテンシ」が無いのに Feature Store（過剰）。逆に**それらが有る**のに BQ テーブル直読み（skew・再利用で不利）。
4. **ストリーミングを見落とす** — `real-time` / `streaming` があるのに BQ バッチ前処理を選ぶ。→ Dataflow。

---

## §5.6 判断フローチャート

**前処理エンジンの判断**

```
ストリーミング（リアルタイム取り込み）が要る？
 ├─ Yes → Dataflow（Beam で stream+batch 統一）
 └─ No → 既存の Spark/Hadoop 資産がある？
     ├─ Yes → Dataproc
     └─ No → データは BigQuery にあり SQL で足りる？
         ├─ Yes → BigQuery 前処理
         └─ No → Dataflow（バッチ）
※ 学習と推論で変換を一致させたい → TFT を併用して skew を防ぐ
```

**特徴量基盤の判断**

```
特徴量を複数チームで再利用 or オンライン低レイテンシ取得が要る？
 ├─ Yes → Vertex AI Feature Store
 └─ No → 学習用に保管・共有するだけ？
     ├─ Yes → BigQuery テーブル
     └─ No（単発試作） → その場前処理（本番では避ける）
```

## §5.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> 複数の ML チームが「直近30日の購入回数」など同じ特徴量を別々に作っており、定義のズレと学習・推論間の不整合（skew）が問題になっている。特徴量を一元管理し、オンライン推論では低レイテンシで取得したい。最適な方法は？
>
> A. 各チームが BigQuery で都度 SQL を書く
> B. **Vertex AI Feature Store で特徴量を一元管理し、オンライン取得を使う**
> C. 学習スクリプト内でその場前処理
> D. CSV を共有ドライブで配布

- **正解: B**。決定打は「**複数チームで再利用**」「**定義のズレ/skew**」「**オンライン低レイテンシ**」（思想⑦組織横断 + skew 防止）。
- A を切る: 都度 SQL は定義のズレと skew を解決しない。
- C を切る: その場処理は再利用も skew 防止もできない。
- D を切る: 統制も低レイテンシ取得もない。

### 問題 2

> IoT センサーから届く**ストリーミングデータ**を、到着と同時に前処理して特徴量化したい。バッチの履歴データも同じロジックで処理したい。最適な前処理基盤は？
>
> A. BigQuery のスケジュールクエリ
> B. Dataproc に Spark Streaming クラスタ
> C. **Dataflow（Apache Beam で stream+batch を同一コード）**
> D. Cloud Functions を多数つなぐ

- **正解: C**。決定打は「**ストリーミング**」「**バッチも同じロジック**」。Beam の統一モデルが適合。
- A を切る: スケジュールクエリはストリーミング即時処理に不向き。
- B を切る: Spark 資産がない前提なら Dataflow のほうがサーバーレスで適切（マネージド優先）。
- D を切る: 関数手組みは統一処理・状態管理が脆弱。

### 問題 3

> あるチームのデータはすべて BigQuery にあり、特徴量化は標準的な集計・正規化で済む。学習は BigQuery ML で行う。追加インフラを増やさず skew も避けたい。最適な方法は？
>
> A. データを GCS にエクスポートして Dataflow で前処理
> B. **BigQuery（SQL / BQML の `TRANSFORM`）で前処理し、学習・推論で同じ変換を適用**
> C. Dataproc クラスタを立てる
> D. Feature Store を新規導入

- **正解: B**。決定打は「**データは BQ**」「**標準的な前処理**」「**追加インフラを増やさず**」（思想②データ所在地）。BQML の `TRANSFORM` 句は学習時の前処理を推論時にも自動適用し skew を防ぐ。
- A / C を切る: 不要なデータ移動・クラスタ管理。
- D を切る: 再利用・オンライン要件がなく過剰。

---

## §5.8 最後に覚える一文

> **ストリーミングは Dataflow、Spark 資産は Dataproc、BQ にあるなら BigQuery 前処理。複数チーム再利用とオンライン取得・skew 防止なら Feature Store。前処理は「学習と推論で同じ場所・同じ変換」に寄せる。**
