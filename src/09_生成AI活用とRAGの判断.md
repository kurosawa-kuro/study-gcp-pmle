# 第9章 生成AI活用とRAGの判断

> 意思決定: **生成AI 要件に対し、既存の基盤モデルを使うか、RAG で自社データに根拠づけるか、どう構築するか。**
> Model Garden / Gemini API / マネージド RAG（Vertex AI Search）/ カスタム RAG（Embeddings + Vector Search）/ Grounding を切り分ける。現行 PMLE が明示的に含む生成AI領域の中心。思想①マネージド優先・②データ所在地が効く。

> ⚠ 要公式確認: 本章の生成AI系サービス名（Model Garden / Vertex AI Search / Agent Builder / RAG Engine / Gemini モデル世代）はリブランド・統廃合が進行中。**現行の正式名称・機能は公式ドキュメントで必ず確認すること**（`doc/02_移行ロードマップ.md` Phase 3 完了条件）。本章は判断軸を主とし、名称は「現行名（旧称）」併記の方針で書く。

---

## §9.1 この章で切れる選択肢

- **Model Garden** — 基盤モデルの選択・カタログ
- **Gemini API（Vertex AI）** — 既存の基盤モデルをそのまま呼ぶ
- **マネージド RAG = Vertex AI Search（旧 Enterprise Search / Agent Builder 系）** — 自社文書の検索拡張を素早く
- **カスタム RAG = Embeddings + Vector Search（+ RAG Engine）** — 独自制御の検索拡張
- **Grounding** — 回答を出典（Google 検索 / 自社データ）に根拠づける
- 対比: 学習する選択肢（Fine-tuning は第10章）

---

## §9.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Model Garden | Gemini や OSS など複数の基盤モデルを選んで使える Vertex AI のカタログ。 |
| Gemini API | マネージドな基盤モデルを API で呼ぶ。プロンプトだけで多くのタスクに対応。 |
| Vertex AI Search | 自社文書をインデックスし、検索 + 根拠づき回答（RAG）をマネージドに提供。低コードで早い。 |
| Embeddings API | テキスト等をベクトル化する。カスタム RAG の検索部品。 |
| Vector Search | 大規模ベクトルの近傍検索（ANN）基盤。カスタム RAG の検索エンジン。 |
| RAG Engine | 取り込み・チャンク化・検索・生成を束ねるマネージド RAG オーケストレーション部品。 |
| Grounding | 生成回答を出典（Google 検索や自社データ）に紐づけ、ハルシネーションを抑える。 |

---

## §9.3 試験で選ぶ条件

### まず「学習せずに済むか」

| 選択 | 選ぶ条件 | 試験キーワード |
|---|---|---|
| Gemini API / Model Garden | 汎用生成（要約・分類・チャット・抽出）/ 既存モデルで足りる | `foundation model`, `summarize`, `chatbot`, `prompt` |
| Model Garden（モデル選択） | 複数モデルを比較・選定したい / OSS も検討 | `model selection`, `compare foundation models` |

### 自社の最新・固有情報に基づいて答えたい → RAG

| 選択 | 選ぶ条件 | 閾値 |
|---|---|---|
| マネージド RAG（Vertex AI Search） | 自社文書の検索拡張を**早く・低コードで** / 運用負荷を下げたい | マネージドで足りるか |
| カスタム RAG（Embeddings + Vector Search） | 検索ロジック・チャンク戦略・ランキングを**独自制御** / 既存ベクトル基盤と統合 | 細かい制御が要るか |
| Grounding | 出典提示・ハルシネーション低減が要件 | 根拠・引用が要るか |

---

## §9.4 試験で選ばない条件

- **Fine-tuning を選ばない（知識追加目的では）**: 「最新情報」「社内文書に基づく」が要件なら、まず RAG。知識の追加・鮮度は RAG の領分で、fine-tuning は不向き（第10章で詳述）。
- **カスタム RAG を選ばない**: マネージド（Vertex AI Search）で足りるのに、Embeddings + Vector Search を自前構築（運用過剰、思想①に反する）。
- **学習を選ばない**: 汎用タスクは Gemini API のプロンプトで足りることが多い。いきなり学習は遠回り。
- **Grounding を省かない**: 「出典を示す」「事実性が重要」なのに素の生成のみ、は誤り。

---

## §9.5 ひっかけパターン

1. **「最新情報 = fine-tuning」短絡** — 社内文書や最新データに基づく回答は **RAG**。fine-tuning はモデルの振る舞い（形式・文体・タスク適応）であって知識の鮮度更新ではない（最頻出の生成AIひっかけ）。
2. **マネージド/カスタムの取り違え** — 「早く・低コードで」はマネージド（Vertex AI Search）、「検索ロジックを独自制御」はカスタム（Embeddings + Vector Search）。
3. **ハルシネーション対策の見落とし** — 「事実性」「出典提示」が要件なのに Grounding/RAG を含まない素の生成。
4. **過剰な自前構築** — 小規模 PoC なのに Vector Search クラスタを自前最適化（マネージドで足りる）。

---

## §9.6 判断フローチャート

```
要件は生成タスク（要約/分類/チャット/抽出）？
 ├─ 自社の固有・最新情報に基づいて答える必要がある？
 │   ├─ Yes → RAG（検索拡張）
 │   │   ├─ 早く・低コード・運用負荷を下げる → Vertex AI Search（マネージド RAG）
 │   │   └─ 検索ロジック/チャンク/ランキングを独自制御 → Embeddings + Vector Search（+ RAG Engine）
 │   │   ※ 出典提示・事実性 → Grounding を併用
 │   └─ No（汎用生成で足りる） → Gemini API / Model Garden（プロンプト）
 └─ モデル自体の振る舞い（文体/形式/専門タスク適応）を変えたい → Fine-tuning（第10章）
```

## §9.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> 社内ヘルプデスク用チャットボットを作る。回答は**常に最新の社内規定文書**に基づき、**出典も提示**したい。運用負荷は最小にしたい。最適な方法は？
>
> A. 社内文書で基盤モデルを fine-tuning する
> B. **Vertex AI Search（マネージド RAG）で社内文書をインデックスし、Grounding で出典付き回答**
> C. Gemini API に毎回全文書をプロンプトで貼る
> D. BigQuery ML で分類

- **正解: B**。決定打は「**最新の社内文書に基づく**」「**出典提示**」「**運用負荷最小**」= マネージド RAG + Grounding（思想①②）。
- A を切る: fine-tuning は知識の鮮度更新に不向き。文書更新のたびに再学習は非現実的。
- C を切る: 文書全部をプロンプトはコンテキスト長・コストで破綻。
- D を切る: 生成タスクで分類器は無関係。

### 問題 2

> 研究チームが、独自のチャンク分割・ハイブリッド検索・リランキングを試したい。既に Vector Search の基盤も持っている。最適な方法は？
>
> A. Vertex AI Search のマネージド設定だけ
> B. **Embeddings + Vector Search でカスタム RAG を構築（必要に応じて RAG Engine）**
> C. fine-tuning
> D. プロンプトだけ

- **正解: B**。決定打は「**独自のチャンク/検索/リランキング**」「**Vector Search 基盤を保有**」= カスタム RAG。
- A を切る: マネージドは細かい制御に限界。要件は独自制御。
- C を切る: 検索戦略の問題は学習では解けない。
- D を切る: 検索拡張が要件でプロンプト単体では不足。

### 問題 3

> 汎用的な問い合わせメールを**要約・分類**したい。社内固有知識は不要で、できるだけ早く始めたい。最適な方法は？
>
> A. **Gemini API にプロンプトで要約・分類させる**
> B. カスタム RAG を構築
> C. AutoML テキストを学習
> D. fine-tuning

- **正解: A**。決定打は「**汎用**」「**固有知識不要**」「**早く**」= 既存基盤モデルをプロンプトで使う。
- B を切る: 固有知識不要なら RAG は過剰。
- C / D を切る: 学習は立ち上げが遅く、汎用タスクには不要。

---

## §9.8 最後に覚える一文

> **汎用生成は Gemini API、最新・社内知識は RAG（早く・低コードなら Vertex AI Search、独自制御なら Embeddings + Vector Search）、事実性・出典は Grounding。「最新情報を入れたい」は fine-tuning ではなく RAG。**
