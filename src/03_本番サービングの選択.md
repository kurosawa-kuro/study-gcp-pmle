# 第3章 本番サービングの選択

> 意思決定: **学習したモデルを、どう推論として提供するか。**
> 推論方式（online / batch）、デプロイ形態（事前ビルド / カスタムコンテナ）、スケール戦略（autoscaling / GPU serving）、デプロイ戦略（traffic split / canary / shadow）を切り分ける。出題ドメイン「モデル提供とスケーリング（想定 20%）」の中心。

---

## §3.1 この章で切れる選択肢

推論方式:
- **Online prediction（エンドポイント）** — 低レイテンシ同期推論
- **Batch prediction** — 大量・非同期のスコアリング

デプロイ形態:
- **事前ビルド済み推論コンテナ** / **カスタムコンテナ（CPR: Custom Prediction Routine 含む）**
- **public endpoint** / **private endpoint（VPC）**

スケール・デプロイ戦略:
- **autoscaling**（min/max replica）、**GPU serving**
- **traffic split / canary / A-B / shadow deployment**
- 比較対象として **GKE（KServe）自前サービング**

---

## §3.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Online prediction | Vertex AI Endpoint にモデルをデプロイし、低レイテンシで 1〜数件をリアルタイム推論する。常時稼働。 |
| Batch prediction | 入力（BQ / GCS）に対し大量データを非同期でまとめて推論する。エンドポイント常駐不要。 |
| 事前ビルド済みコンテナ | TF / sklearn / XGBoost 等の Google 提供推論コンテナ。モデル成果物を置くだけ。 |
| カスタムコンテナ / CPR | 前処理・後処理や独自ランタイムを推論に含めたいときに使う。 |
| private endpoint | エンドポイントを VPC 内に閉じる。公開せず低レイテンシ・セキュア。 |
| autoscaling | トラフィックに応じて replica を増減。`min_replica_count` / `max_replica_count` で制御。 |
| traffic split | 1 エンドポイント内で複数モデルにトラフィックを%配分（A-B / canary に使う）。 |

> ⚠ 要公式確認: スケール下限の挙動（Vertex Endpoint の `min_replica_count` と「ゼロへの縮退」可否）や co-hosting / 共有リソースプールの仕様は変化し得る。現行仕様は公式ドキュメントで確認すること。

---

## §3.3 試験で選ぶ条件

### Online vs Batch（最重要分岐）

| 方式 | 選ぶ条件 | 試験キーワード | 閾値 |
|---|---|---|---|
| Online prediction | リアルタイム / 低レイテンシ / 1 件ごと / ユーザー対面 | `real-time`, `low latency`, `synchronous`, `per-request`, `online` | レイテンシ要件があるか |
| Batch prediction | 夜間バッチ / 数百万件 / 即時性不要 / 定期スコアリング | `nightly`, `scheduled`, `millions of records`, `asynchronous`, `offline scoring` | 即時性が不要で大量か |

> 第一分岐: **「レイテンシ要件があるか / 即時に結果が要るか」**。要るなら online、不要で大量なら batch。

### デプロイ形態

| 形態 | 選ぶ条件 |
|---|---|
| 事前ビルド済みコンテナ | 標準フレームワーク・標準入出力 |
| カスタムコンテナ / CPR | 前処理・後処理を推論に含めたい / 独自依存 |
| private endpoint | VPC 内限定・公開不可・社内システム連携 |
| public endpoint | 外部公開が要件 |

### スケール・デプロイ戦略

| 戦略 | 選ぶ条件 |
|---|---|
| autoscaling（min/max） | トラフィック変動に追随しつつコスト最適化 |
| min_replica >= 1 常駐 | cold start を避けたい / 常にレイテンシ保証が要る |
| GPU serving | 大型 DL モデルで CPU では推論レイテンシが足りない |
| traffic split / canary | 新モデルを段階的に出して比較・リスク低減 |
| shadow deployment | 本番トラフィックを新モデルにも流すが応答は返さず比較のみ |

---

## §3.4 試験で選ばない条件

- **Online を選ばない**: 即時性が不要な大量スコアリングなのに常時エンドポイント（コスト過剰）。→ Batch。
- **Batch を選ばない**: ユーザー対面の低レイテンシ要件なのにバッチ（要件未達）。→ Online。
- **GKE/KServe 自前を選ばない**: マネージドで足りるのに Kubernetes を自前運用（思想①マネージド優先に反する）。**ただし**既存 GKE 資産・きめ細かい制御・scale-to-zero が要件なら KServe が正解になることもある。
- **min_replica を上げすぎない**: トラフィックが小さいのに常時多数 replica（コスト過剰）。

---

## §3.5 ひっかけパターン

1. **「ML 推論 = エンドポイント」短絡** — 夜間に数百万件を一括スコアリングするだけなのに online endpoint を選ばせる。→ Batch prediction（コストと用途で正解）。
2. **「リアルタイム」を見落とす** — `low latency` / `real-time` があるのに batch を選ぶ。→ Online。
3. **前処理の置き場所** — クライアント側で複雑な前処理をさせる選択肢。→ 学習と推論で前処理を揃えるためカスタムコンテナ/CPR にまとめ、training-serving skew を防ぐ（思想③本番運用前提）。
4. **コスト最適化の取り違え** — 低トラフィックでレイテンシ厳しくないのに常時大規模常駐。→ autoscaling（min を低く）。逆に**レイテンシ保証必須**なら `min_replica >= 1` で cold start を避ける（trade-off を読む）。

---

## §3.6 判断フローチャート

**推論方式の判断**

```
即時に結果が要る／レイテンシ要件がある？
 ├─ Yes → Online prediction（Vertex AI Endpoint）
 │   ├─ 前処理・後処理を含めたい → カスタムコンテナ / CPR
 │   ├─ VPC 内限定 → private endpoint
 │   └─ レイテンシ保証必須 → min_replica >= 1（cold start 回避）
 └─ No（大量・定期・非同期） → Batch prediction（BQ/GCS 入出力）
```

**デプロイ戦略の判断**

```
新モデルを本番に出す。リスクを抑えたい？
 ├─ 段階的に比較しながら切替 → traffic split / canary（%配分）
 ├─ 応答は旧のまま、新モデルの挙動だけ観測 → shadow deployment
 └─ 既存 GKE 資産あり／scale-to-zero／細かい制御が必須 → GKE + KServe
     （それ以外はマネージドの Vertex Endpoint を優先）
```

## §3.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> EC サイトが、ユーザーが商品ページを開いた瞬間に**レコメンドをリアルタイム表示**したい。応答は数十ミリ秒以内が必須。最適なサービング方式は？
>
> A. Batch prediction を 1 時間ごとに実行
> B. **Vertex AI Online prediction エンドポイント（必要に応じて min_replica >= 1）**
> C. BigQuery に夜間で予測を書き込む
> D. ローカルでモデルを実行

- **正解: B**。決定打は「**リアルタイム表示**」「**数十ミリ秒以内**」（思想⑥スケール分岐・レイテンシ要件）。cold start を避けるため min_replica を確保。
- A / C を切る: バッチは即時性ゼロ。レイテンシ要件未達。
- D を切る: スケール・運用面で本番要件を満たさない。

### 問題 2

> 与信スコアリングを、毎晩 1 回・全顧客 2,000 万件に対してまとめて計算し、結果を BigQuery に書き戻したい。即時性は不要でコストを抑えたい。最適な方式は？
>
> A. Online エンドポイントに 1 件ずつ送る
> B. **Batch prediction（入力・出力ともに BigQuery）**
> C. 常時 50 replica のエンドポイント
> D. Dataflow でルールベース計算

- **正解: B**。決定打は「**毎晩 1 回**」「**2,000 万件**」「**即時性は不要**」「**コストを抑えたい**」（バッチの典型）。
- A を切る: 2,000 万件を 1 件ずつ online は非効率・高コスト。
- C を切る: 常時大量常駐はコスト過剰。
- D を切る: 学習済み ML モデルでの推論が要件で、ルールベースではない。

### 問題 3

> 新しいレコメンドモデルを本番投入したいが、いきなり 100% 切り替えるのは怖い。まず 5% のトラフィックで指標を比較し、問題なければ徐々に増やしたい。最適な方法は？
>
> A. shadow deployment
> B. **同一エンドポイントの traffic split で新モデルに 5% 配分（canary）**
> C. 別エンドポイントを作り DNS で振り分け
> D. Batch prediction で検証

- **正解: B**。決定打は「**5% から徐々に**」「**指標を比較**」= canary。Vertex Endpoint の traffic split が直接対応。
- A を切る: shadow は応答を返さず観測のみ。「徐々に切り替える」要件と異なる（紛らわしい）。
- C を切る: DNS 振り分けは自前運用でマネージド優先に反する。
- D を切る: バッチは本番トラフィックでの段階切替に使えない。

---

## §3.8 最後に覚える一文

> **レイテンシ要件があれば Online エンドポイント、大量・非同期なら Batch prediction。前処理はカスタムコンテナ/CPR にまとめて skew を防ぐ。段階切替は traffic split（canary）、観測だけなら shadow。マネージドで足りるなら GKE 自前は選ばない。**
