# 序章 試験の世界観と本書の読み方

> 本書は「技術を理解する本」ではなく、**問題文の制約条件から正解サービス・構成・運用判断を選ぶための判断辞書**である（`doc/01_仕様と設計.md §1` 参照）。
> 序章では、各章の判断の土台となる「PMLE の世界観（正解になりやすい設計思想）」と、本書の使い方を示す。

---

## §0.1 PMLE はどういう試験か

PMLE（Google Cloud Professional Machine Learning Engineer）は、**ML アルゴリズムの知識を問う試験ではなく、「この制約なら Google Cloud 上でどの構成を選ぶか」を問う設計判断の試験**である。

- 出題は「これは何ですか？」より「**この条件なら何を選ぶべきか？**」が圧倒的に多い。
- 選択肢には似たサービスが並ぶ。よって「A を知っているか」より「**A と B を切り分けられるか**」が合否を分ける。
- コーディング試験ではない。Python / SQL の最低限の読解でコードスニペットを解釈できればよく、実装力そのものは直接評価されない。

> ⚠ 要公式確認: 出題ドメインと比率は改訂され得る。本書執筆時点の想定比率は次のとおり（`doc/01 §1` の値）。試験前に [現行の公式試験ガイド](https://cloud.google.com/learn/certification/machine-learning-engineer) で必ず再確認すること。
>
> | 出題ドメイン | 想定比率 | 本書の重み |
> |---|---|---|
> | ML パイプラインの自動化とオーケストレーション | 22% | 最重点（第4章） |
> | モデル提供とスケーリング | 20% | 最重点（第3章） |
> | プロトタイプの ML モデルへのスケーリング | 18% | 重点（第2章） |
> | 低コード ML ソリューションの設計 | — | 重点（第1章） |
> | データとモデルの管理（チーム横断） | — | Phase 2 |
> | ML ソリューションの監視 | — | Phase 2 |

## §0.2 正解になりやすい 7 つの設計思想

PMLE には「Google Cloud 試験らしい思想」がある。**迷ったらこの思想に寄せた選択肢が正解になりやすい**。各章の「選ぶ条件」はすべてこの思想から導かれる。

| # | 思想 | 試験上の意味 | 典型的な現れ方 |
|---|---|---|---|
| 1 | **マネージド優先** | 自前構築より Vertex AI / BigQuery / AutoML を優先しやすい | 「運用負荷を下げたい」「インフラ管理をしたくない」→ マネージド側が正解 |
| 2 | **既存データ所在地を尊重** | データが BigQuery にあるなら BigQuery ML / BigQuery 連携を疑う | 「データは既に BigQuery にある」→ データを動かさない選択が正解 |
| 3 | **本番運用前提** | prototype だけでなく monitoring / retraining / lineage まで考える | 「本番にデプロイした後」→ 監視・再学習・系譜まで含む選択肢 |
| 4 | **再現性重視** | notebook 単発ではなく pipeline 化・metadata 管理 | 「再現できるように」「チームで共有」→ Pipelines / Experiments |
| 5 | **責任ある AI** | bias / fairness / explainability / privacy が出たら軽視しない | これらの語が出たら必ず対応する選択肢が正解。無視する選択肢は誤り |
| 6 | **スケール要件で分岐** | batch / online、CPU / GPU / TPU、endpoint / container を切る | 「リアルタイム」「夜間バッチ」「数百万件」→ スケール語で分岐 |
| 7 | **組織横断利用** | Feature Store / Model Registry / Experiments が出やすい | 「複数チームで再利用」「特徴量を共有」→ 共有基盤が正解 |

> この 7 思想は、各章の判断フローチャートの「第一分岐」になっていることが多い。困ったら「どの思想が問われているか」を先に特定する。

## §0.3 本書の構成と読み方

### 章の立て方

章は**サービス名ではなく意思決定単位**で立てている。「BigQuery ML 章」ではなく「モデル構築方式の選択」という形にすることで、「A と B の切り分け」が学べるようにしている。

| 章 | 問われる意思決定 | 主なドメイン |
|---|---|---|
| [第1章 モデル構築方式の選択](01_モデル構築方式の選択.md) | どの方式でモデルを作るか（BQML / AutoML / Custom） | 低コード + プロトタイプ |
| [第2章 カスタム学習を本番にスケールする](02_カスタム学習を本番にスケールする.md) | プロトタイプをどう本番学習にスケールするか | プロトタイプのスケーリング 18% |
| [第3章 本番サービングの選択](03_本番サービングの選択.md) | どう推論を提供するか（online / batch / container） | モデル提供 20% |
| [第4章 ML パイプラインの自動化とオーケストレーション](04_MLパイプラインの自動化とオーケストレーション.md) | どうワークフローを自動化・連携するか | パイプライン自動化 22% |

> Phase 2 以降で追加する章（運用監視・データ/特徴量管理・生成AI・責任あるAI 等）は [`doc/02_移行ロードマップ.md`](../doc/02_移行ロードマップ.md) を参照。

### 各章の 8 項目テンプレート

全章は次の 8 項目で統一されている（定義は `doc/01_仕様と設計.md §2.1` が正）。**本文と判断材料は分離していない** — 本文中に常に「選ぶ / 除外 / ひっかけ」が埋め込まれている。

1. **この章で切れる選択肢** — 章で切り分ける候補サービス
2. **技術の最短定義** — 各サービス 3〜5 行
3. **試験で選ぶ条件** — 問題文のキーワード / 制約 / 正解になりやすい文脈
4. **試験で選ばない条件** — 競合に負ける条件 / 自前構築すべき条件 / 過剰設計
5. **ひっかけパターン** — よくある誤選択 / 問題文の罠
6. **判断フローチャート** — 3〜7 分岐で即断する
7. **代表問題パターン** — 自作シナリオ。正解 / 各不正解を切る理由 / 決定打の語
8. **最後に覚える一文** — その章のエッセンス

### 付録の使い方

- [付録A シグナル語辞典](90_付録_シグナル語辞典.md) — 問題文の表現から疑うべき選択肢を逆引きする。**解くときの最初の一手**。
- [付録B 比較表](91_付録_比較表.md) — 似たサービスの閾値を一覧化したチートシート。**直前の最終確認**に使う。

## §0.4 解き方の基本動作

1. 問題文を読み、**シグナル語**（`low latency`, `BigQuery tables`, `nightly`, `RAG` 等）を拾う → [付録A](90_付録_シグナル語辞典.md) で疑う選択肢を引く。
2. **7 思想**（§0.2）のどれが問われているかを特定する（マネージド優先か、データ所在地か、本番運用か…）。
3. 候補が 2 つに絞れたら、その章の**比較表の「閾値」**で切る（例: 「BigQuery 内で完結できるか？」）。
4. 残った選択肢を**ひっかけパターン**（各章 §N.5）と照合し、罠でないか確認する。

## §0.5 最後に覚える一文

> **PMLE は「制約 → どの構成を選ぶか」を問う試験。迷ったらマネージド・データ所在地・本番運用前提に寄せる。**
