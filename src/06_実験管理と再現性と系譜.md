# 第6章 実験管理・再現性・系譜

> 意思決定: **学習の試行をどう記録・比較し、成果物の系譜（lineage）と再現性をどう担保し、モデルをどう版管理するか。**
> Experiments / ML Metadata / Model Registry / TensorBoard を切り分ける。出題ドメイン「ML パイプライン自動化」「データとモデルの管理」に直結し、思想④再現性が核。

---

## §6.1 この章で切れる選択肢

- **Vertex AI Experiments** — 試行（run）のパラメータ・指標を記録・比較
- **Vertex ML Metadata** — アーティファクト・実行・**系譜（lineage）**を記録
- **Vertex AI Model Registry** — モデルの**版管理**・エイリアス・デプロイ起点
- **Vertex AI TensorBoard** — 学習曲線などの**可視化**

---

## §6.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Vertex AI Experiments | 複数の学習 run のハイパーパラメータ・メトリクス・成果物を記録し、横並びで比較する。 |
| Vertex ML Metadata | パイプラインの入出力アーティファクト・実行・依存関係を記録し、「どのデータ→どの学習→どのモデル」の系譜をたどれる。Vertex AI Pipelines が自動記録。 |
| Model Registry | 学習済みモデルをバージョン管理し、エイリアス（例 `default` / `prod`）を付けてデプロイの起点にする。 |
| TensorBoard | 損失・精度などのメトリクスや埋め込みを時系列で可視化する。 |

---

## §6.3 試験で選ぶ条件

| 要素 | 選ぶ条件 | 試験キーワード |
|---|---|---|
| Experiments | 複数試行の比較 / どのハイパラが良かったか / チームで共有 | `compare runs`, `track experiments`, `hyperparameter comparison` |
| ML Metadata | 系譜・監査・「どのデータから来たモデルか」/ 再現性 | `lineage`, `artifact tracking`, `audit`, `reproducibility` |
| Model Registry | モデルの版管理 / デプロイ対象の切替 / 承認フロー | `model versioning`, `model registry`, `promote to production`, `aliases` |
| TensorBoard | 学習の可視化 / 収束の確認 / 埋め込みの可視化 | `visualize training`, `learning curves`, `TensorBoard` |

---

## §6.4 試験で選ばない条件

- **手動記録（スプレッドシート等）を選ばない**: 「再現性」「比較」「監査」が出たら手作業は誤り。
- **Experiments と Registry を取り違えない**: 「試行の比較」は Experiments、「モデルの版管理・デプロイ起点」は Registry。
- **ML Metadata を省かない**: 「系譜」「どのデータ由来か」「監査」が要件なのに含まない選択肢は誤り。
- **TensorBoard を万能視しない**: TensorBoard は可視化であって、版管理・系譜・デプロイ管理はしない。

---

## §6.5 ひっかけパターン

1. **Experiments vs Registry の混同** — 「本番に昇格するモデルを管理」は Registry、「どの設定が良いか比較」は Experiments。**run の比較か、モデルの版管理か**で切る。
2. **再現性を可視化で代替** — 「再現できるように」に対し TensorBoard を選ばせる。→ 可視化では再現性は担保できない。Pipelines + ML Metadata（系譜）が正解。
3. **手動運用への誘導** — ノートブックで結果をメモする選択肢。→ 思想④再現性に反する。
4. **系譜の欠落** — 監査要件があるのに成果物追跡（ML Metadata）を含まないパイプライン。

---

## §6.6 判断フローチャート

```
何を解決したい？
 ├─ 複数の学習試行を比較したい → Vertex AI Experiments
 ├─ 学習の収束・メトリクスを可視化したい → Vertex AI TensorBoard
 ├─ どのデータ→どの学習→どのモデルか（系譜・監査・再現性） → Vertex ML Metadata
 │   （Vertex AI Pipelines を使えば自動記録）
 └─ 学習済みモデルを版管理し本番デプロイの起点にしたい → Model Registry
```

## §6.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> 監査要件のあるチームが「本番モデルが、どのデータセット・どの前処理・どの学習ジョブから生成されたか」を後から追跡できるようにしたい。手作業は避けたい。最適な方法は？
>
> A. 学習ログをスプレッドシートに記録
> B. **Vertex AI Pipelines で学習を実行し、ML Metadata に系譜を自動記録する**
> C. TensorBoard で可視化
> D. モデルファイルに命名規則を付けて GCS に保存

- **正解: B**。決定打は「**どのデータ→どのモデルか**」「**監査**」「**手作業を避ける**」= 系譜（lineage）。Pipelines + ML Metadata が自動記録（思想④）。
- A / D を切る: 手作業・命名規則は再現性・監査に耐えない。
- C を切る: TensorBoard は可視化で系譜は追えない。

### 問題 2

> 50 通りのハイパーパラメータ設定で学習し、**どの組合せが最も良い指標を出したか**をチームで横並び比較したい。最適な方法は？
>
> A. **Vertex AI Experiments で各 run のパラメータと指標を記録し比較**
> B. Model Registry にすべて登録
> C. 各 run を別々のノートブックで実行
> D. Cloud Logging を grep

- **正解: A**。決定打は「**どの組合せが良いか**」「**横並び比較**」= Experiments。
- B を切る: Registry は版管理であって試行比較の道具ではない。
- C / D を切る: 比較が手作業・非構造で再現性が低い。

### 問題 3

> 学習済みモデルを複数バージョン保持し、`prod` エイリアスが指すバージョンだけを本番エンドポイントにデプロイ・切替したい。最適な方法は？
>
> A. GCS のフォルダ名でバージョン管理
> B. Experiments にモデルを保存
> C. **Vertex AI Model Registry でバージョン管理しエイリアスでデプロイ対象を制御**
> D. 毎回再学習する

- **正解: C**。決定打は「**複数バージョン**」「**エイリアスでデプロイ切替**」= Model Registry。
- A を切る: フォルダ名は版管理・昇格フローに弱い。
- B を切る: Experiments は試行記録でモデル版管理ではない。
- D を切る: 再学習は版管理の解にならない。

---

## §6.8 最後に覚える一文

> **試行の比較は Experiments、可視化は TensorBoard、系譜・監査・再現性は ML Metadata（Pipelines が自動記録）、モデルの版管理とデプロイ起点は Model Registry。再現性は可視化では代替できない。**
