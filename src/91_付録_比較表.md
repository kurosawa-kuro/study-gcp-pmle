# 付録B 比較表（チートシート）

> 似たサービスの**閾値**を一覧化した最終確認用チートシート。直前の見直しに使う（序章 §0.4）。
> 出典の比較軸は `doc/01_仕様と設計.md §2.2`。「必須度」はその軸の出題重要度。

## B.1 BigQuery ML vs AutoML vs Custom Training（必須度: 最重要 → [第1章](01_モデル構築方式の選択.md)）

| 観点 | BigQuery ML | AutoML | Custom Training |
|---|---|---|---|
| 入力 | BQ の構造化データ | 表形式/画像/テキスト + ラベル | 任意（自分のコード次第） |
| 必要スキル | SQL | ほぼ不要（低コード） | ML エンジニアリング |
| 制御度 | 低（SQL の範囲） | 低（自動） | 高（アーキ/損失/分散） |
| 速さ（立ち上げ） | 速い | 速い | 遅い（実装が要る） |
| 第一分岐 | BQ 内で完結できるか | 低コードで高品質が要るか | 独自コード/制御が要るか |

## B.2 Vertex AI Pipelines vs Kubeflow vs Cloud Composer（必須度: 最重要 → [第4章](04_MLパイプラインの自動化とオーケストレーション.md)）

| 観点 | Vertex AI Pipelines | Kubeflow Pipelines (KFP) | Cloud Composer |
|---|---|---|---|
| 主目的 | ML パイプラインのマネージド実行 | パイプライン定義（DAG/コンポーネント） | 複数システム横断のワークフロー |
| 実行基盤 | サーバーレス（Google 管理） | Vertex Pipelines 上で実行が定番 | マネージド Airflow（環境を持つ） |
| 系譜記録 | ML Metadata に自動 | （Vertex 上で）自動 | 自前で設計 |
| 向くスコープ | ML 単体（前処理→学習→デプロイ） | ML 単体（再利用コンポーネント） | ETL + 外部連携 + ML の全体統合 |
| 第一分岐 | ML 単体か | コンポーネント定義をどう書くか | 複数システム横断か |

## B.3 Online vs Batch prediction（必須度: 最重要 → [第3章](03_本番サービングの選択.md)）

| 観点 | Online prediction | Batch prediction |
|---|---|---|
| レイテンシ | 低（同期・即時） | 高くてよい（非同期） |
| 入力単位 | 1〜数件/リクエスト | 大量（BQ/GCS の一括） |
| 稼働形態 | 常時エンドポイント | 都度ジョブ（常駐不要） |
| コスト傾向 | 常駐分かかる | 実行時のみ |
| 第一分岐 | レイテンシ要件があるか | 即時不要で大量か |

## B.4 Model Monitoring vs Explainable AI vs Evaluation（必須度: 最重要 → [第7章](07_モデル運用監視の判断.md) / [第8章](08_責任あるAIと説明可能性と評価.md)）

| 観点 | Model Monitoring | Explainable AI | Model Evaluation |
|---|---|---|---|
| 問い | 本番で入力/予測が変化したか | なぜこの予測になったか | モデルはどれだけ良いか |
| 主対象 | training-serving skew / drift | feature attribution | 指標（精度/AUC/RMSE 等） |
| タイミング | 本番運用中（継続） | 学習後/運用中 | 学習後/比較時 |
| 決定打の語 | drift / skew / 継続監視 | attribution / なぜ / fairness | metrics / 比較 / どのモデル |

> skew は baseline=学習データ、drift は baseline=過去本番（[第7章](07_モデル運用監視の判断.md)）。

## B.5 Feature Store vs BigQuery table vs ad hoc 前処理（必須度: 高 → [第5章](05_データと特徴量の管理.md)）

| 観点 | Feature Store | BigQuery table | ad hoc 前処理 |
|---|---|---|---|
| 主目的 | 特徴量の共有・再利用・serving 接続 | 分析/学習データの保管 | その場の変換 |
| skew 防止 | 学習/推論で同じ特徴量を供給 | 設計次第 | 弱い（再現困難） |
| 組織横断 | 強い（チーム間再利用） | 中 | 弱い |
| 決定打の語 | feature reuse across teams | データ所在地/SQL | （避けるべき選択肢になりやすい） |

## B.6 Workbench vs Colab Enterprise vs Dataproc notebook（必須度: 高 → [第2章](02_カスタム学習を本番にスケールする.md)）

| 観点 | Vertex AI Workbench | Colab Enterprise | Dataproc notebook |
|---|---|---|---|
| 主目的 | 本格的な ML 試作 | 軽量・共有の共同編集 | Spark 上の分散処理 |
| 統合 | Vertex/BQ/GCS | IAM 管理された Colab | Hadoop/Spark エコシステム |
| 決定打の語 | managed notebook / Vertex 統合 | shared / collaboration | Spark / Hadoop |

## B.7 Dataflow vs Dataproc vs BigQuery 前処理（必須度: 高 → [第5章](05_データと特徴量の管理.md)）

| 観点 | Dataflow（Beam） | Dataproc（Spark/Hadoop） | BigQuery 前処理 |
|---|---|---|---|
| 主目的 | ストリーミング/バッチの統一処理 | 既存 Spark/Hadoop 資産の移行 | SQL で完結する変換 |
| マネージド度 | 高（サーバーレス） | クラスタ管理あり | 最高（BQ 内） |
| 決定打の語 | streaming / Beam / 統一処理 | existing Spark jobs | data は BQ / SQL |

## B.8 Model Garden vs Agent Builder vs Custom RAG（必須度: 高 → [第9章](09_生成AI活用とRAGの判断.md)）

> ⚠ 要公式確認: 生成AI系はリブランド進行中。現行公式名・機能は要確認（[第9章](09_生成AI活用とRAGの判断.md) 冒頭の flag 参照）。

| 観点 | Model Garden | Vertex AI Search（旧 Agent Builder） | Custom RAG |
|---|---|---|---|
| 主目的 | 基盤モデルの選択/利用 | マネージドな検索拡張・エージェント | 独自制御の RAG 構築 |
| 制御度 | 低〜中 | 中（マネージド） | 高（Embeddings + Vector Search） |
| 決定打の語 | foundation model selection | enterprise search / grounded | 独自制御が必要 |

## B.9 GPU vs TPU vs CPU（必須度: 中 → [第2章](02_カスタム学習を本番にスケールする.md)）

| 観点 | CPU | GPU | TPU |
|---|---|---|---|
| 向くワークロード | 表形式/古典 ML/小規模 | DL 一般（PyTorch 含む）/中規模 | 大規模 TF/JAX/行列演算支配 |
| フレームワーク | 何でも | TF/PyTorch 等 | TF/JAX 中心 |
| 決定打の語 | コスト最優先/小規模 | deep learning | large TF/JAX |

## B.10 A/B test vs canary vs shadow（必須度: 中 → [第3章](03_本番サービングの選択.md)）

| 観点 | A/B test | canary | shadow |
|---|---|---|---|
| 目的 | 2 案を本番比較 | 新版を少しずつ拡大 | 新版を観測のみ（応答は旧） |
| トラフィック | 分割して両方に応答 | 小→大へ段階配分 | 複製して新版にも流すが応答返さない |
| 実現 | traffic split | traffic split（段階） | shadow deployment |
| 決定打の語 | 比較したい | 5% から徐々に | リスクゼロで挙動観測 |

## B.11 Explainable AI: 手法の使い分け（必須度: 高 → [第8章](08_責任あるAIと説明可能性と評価.md)）

| 手法 | 適用モデル | 用途 | 決定打の語 |
|---|---|---|---|
| Sampled Shapley | 微分不可能（ツリー/アンサンブル/表形式） | feature attribution | 表形式 / ツリー |
| Integrated Gradients | 微分可能（ニューラルネット） | feature attribution | NN / 微分可能 |
| XRAI | 画像 | 寄与領域のハイライト | 画像 / どの領域 |

> ⚠ 要公式確認: 手法とモデル種別の対応は変化し得る。現行は公式ドキュメントで確認。

## B.12 生成AI 品質向上の手段（必須度: 高 → [第9章](09_生成AI活用とRAGの判断.md) / [第10章](10_生成AIチューニングと評価.md)）

| 手段 | 解決するもの | 主目的 | 決定打の語 |
|---|---|---|---|
| プロンプト設計 | 指示の与え方 | まず最初・低コスト | prompt / few-shot |
| RAG | **知識**（最新/社内固有） | 鮮度・事実性 | latest / internal docs / grounded |
| Fine-tuning（SFT） | **振る舞い**（文体/形式/専門タスク） | 出力の一貫性・適応 | consistent style/format |
| 蒸留 | コスト/レイテンシ | 小型化 | reduce cost/latency / smaller |

> 最重要: 「最新/社内知識」は **RAG**、「文体/形式/振る舞い」は **fine-tuning**。知識か振る舞いかで切る。

## B.13 セキュリティ・コスト早見（必須度: 中 → [第11章](11_データ所在地とコストとセキュリティ.md)）

| 要件 | 選ぶもの |
|---|---|
| データを region/国に留める | データ所在地（region 固定） |
| 持ち出し（exfiltration）防止の境界 | VPC Service Controls |
| 公開せず VPC 内限定 | private endpoint |
| 暗号鍵を自社管理 | CMEK |
| 鍵を配らず外部から接続 | Workload Identity Federation |
| 即時不要・大量のコスト削減 | batch prediction |
| アイドル多いコスト削減 | scale-to-zero / autoscaling |
| 重い推論のコスト削減 | 小型モデル / 蒸留 |
