# 第7章 モデル運用監視の判断

> 意思決定: **本番モデルの劣化を、何で・どの基準で検知し、いつ再学習するか。**
> training-serving skew / prediction drift / feature attribution drift を切り分け、Model Monitoring・Explainable AI・Evaluation の役割の違いを押さえる。出題ドメイン「ML ソリューションの監視」「モデル提供とスケーリング」に直結。思想③本番運用前提が核。

---

## §7.1 この章で切れる選択肢

- **Vertex AI Model Monitoring** — 本番の入力/予測の変化を継続検知
  - **training-serving skew 検知**
  - **prediction / feature drift 検知**
  - **feature attribution drift 検知**
- 役割の違いで対比:
  - **Explainable AI** — なぜその予測か（第8章）
  - **Model Evaluation** — どれだけ良いか（第8章）
- 検知後のアクション: **アラート → 再学習トリガ（CT）**（第4章のオーケストレーションと接続）

---

## §7.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Model Monitoring | デプロイ済みモデルの入力特徴量や予測の分布を監視し、基準からの逸脱を検知してアラートする。 |
| training-serving skew | **学習データ**と**本番の入力**の分布差。baseline は学習データ。 |
| prediction / feature drift | **本番入力が時間とともに**変化すること。baseline は直近〜過去の本番データ。 |
| feature attribution drift | 各特徴量の**重要度（寄与）**が時間とともに変化すること（Explainable AI と連携）。 |

> ⚠ 要公式確認: Model Monitoring の世代（旧 Endpoint 紐付け型と新しいモデル単位の監視）、対応する検知タイプ・baseline 指定方法・対応モデル種別は変化し得る。現行仕様は公式ドキュメントで確認すること。

---

## §7.3 試験で選ぶ条件

| 検知タイプ | 選ぶ条件（決定打の語） | baseline |
|---|---|---|
| training-serving skew | デプロイ直後から本番入力が学習と違う / 前処理の不一致 | 学習データ |
| prediction / feature drift | 時間経過で入力・予測の傾向が変化（季節性・市場変化） | 過去の本番データ |
| feature attribution drift | 「どの特徴が効いているか」が変化した | 過去の attribution |

監視の設計:

| 観点 | 選ぶ条件 |
|---|---|
| アラート閾値の設定 | しきい値超過で通知（Cloud Monitoring / 通知チャネル） |
| サンプリング率 | 高トラフィックでコストと検知感度のバランス |
| 再学習トリガ | drift/skew 検知 → パイプライン再実行（第4章 Eventarc / スケジュール） |

---

## §7.4 試験で選ばない条件

- **Explainable AI を監視代わりにしない**: 「分布がズレたか」は Monitoring。XAI は「なぜこの予測か」。
- **Evaluation を監視代わりにしない**: Evaluation は学習後/比較時の品質測定。本番の継続検知ではない。
- **監視なしで本番化しない**: 「本番にデプロイした後」が出たら監視・再学習を含む選択肢が正解（思想③）。
- **skew と drift を取り違えない**: 学習 vs 本番の差は skew、本番の時間変化は drift。

---

## §7.5 ひっかけパターン

1. **skew と drift の混同** — 「デプロイ直後から精度が低い」= training-serving skew（学習と本番の前処理/分布の差）。「数ヶ月かけて徐々に劣化」= drift（時間変化）。**baseline が学習データか過去本番データか**で切る。
2. **Monitoring / XAI / Evaluation の混同** — 「分布の逸脱を継続検知」は Monitoring、「予測理由」は XAI、「指標で良し悪し」は Evaluation（`doc/01 §2.2` 最重要比較軸）。
3. **検知して終わり** — drift を検知するだけで再学習につなげない選択肢。→ アラート → 再学習パイプライン（CT）まで（第4章接続）。
4. **過検知/過小検知** — 閾値・サンプリングを無視した選択肢。コストと感度の調整が要る。

---

## §7.6 判断フローチャート

**何を監視すべきか**

```
問題の性質は？
 ├─ デプロイ直後から学習時と挙動が違う → training-serving skew 検知（baseline=学習データ）
 ├─ 時間経過で入力/予測の傾向が変化 → prediction/feature drift 検知（baseline=過去本番）
 └─ 効いている特徴が変わった → feature attribution drift 検知
→ いずれも閾値超過でアラート → 再学習パイプライン（CT）を起動
```

**役割の切り分け（最重要）**

```
問いは？
 ├─ 本番で分布がズレたか（継続監視） → Model Monitoring
 ├─ なぜこの予測になったか → Explainable AI（第8章）
 └─ モデルはどれだけ良いか（指標） → Model Evaluation（第8章）
```

## §7.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> あるレコメンドモデルは**デプロイ直後から**オフライン評価より明らかに精度が低い。調査すると、本番の前処理が学習時と異なっていた。今後これを自動検知したい。最適な方法は？
>
> A. prediction drift 監視（baseline=過去本番）
> B. **training-serving skew 検知（baseline=学習データ）**
> C. Model Evaluation を毎日実行
> D. Explainable AI を有効化

- **正解: B**。決定打は「**デプロイ直後から**」「**学習時と前処理が違う**」= training-serving skew。baseline は学習データ。
- A を切る: drift は時間変化が対象で、直後からの不一致は skew。
- C を切る: Evaluation は品質測定で、入力分布の継続検知ではない。
- D を切る: XAI は理由説明で、分布ズレ検知ではない。

### 問題 2

> 与信モデルが**数ヶ月かけて徐々に**精度低下している。市場環境の変化で入力分布が変わったらしい。劣化を検知して再学習を自動起動したい。最適な方法は？
>
> A. **prediction/feature drift 検知 → 閾値超過で再学習パイプラインを起動**
> B. training-serving skew 検知のみ
> C. TensorBoard で可視化
> D. 手動で毎月再学習

- **正解: A**。決定打は「**数ヶ月かけて徐々に**」「**入力分布が変化**」= drift。検知 → 再学習（CT）まで含むのが本番運用（思想③、第4章接続）。
- B を切る: 時間変化は skew ではなく drift。
- C を切る: 可視化は継続自動検知にならない。
- D を切る: 手動は自動検知・自動再学習の要件に反する。

### 問題 3

> チームは「Model Monitoring」「Explainable AI」「Model Evaluation」の使い分けを問われた。**本番デプロイ後に入力分布の逸脱を継続的にアラート**したいのはどれか？
>
> A. Model Evaluation
> B. Explainable AI
> C. **Model Monitoring**
> D. TensorBoard

- **正解: C**。決定打は「**本番デプロイ後**」「**継続的にアラート**」「**入力分布の逸脱**」。
- A を切る: Evaluation は品質指標の測定（学習後/比較）。
- B を切る: XAI は予測理由の説明。
- D を切る: TensorBoard は学習の可視化。

---

## §7.8 最後に覚える一文

> **学習 vs 本番の差は skew（baseline=学習データ）、本番の時間変化は drift（baseline=過去本番）。検知したら再学習（CT）まで自動化する。Monitoring=ズレ検知、XAI=理由、Evaluation=良し悪し。**
