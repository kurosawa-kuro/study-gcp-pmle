# 第4章 ML パイプラインの自動化とオーケストレーション

> 意思決定: **ML ワークフローをどう自動化し、何でオーケストレーションするか。**
> パイプラインエンジン（Vertex AI Pipelines / Kubeflow / TFX）、ワークフロー全体の連携（Cloud Composer）、起動契機（Scheduler / Eventarc）、CI/CD（Cloud Build）を切り分ける。出題ドメイン「ML パイプラインの自動化とオーケストレーション（想定 22%）」= 最大配点の中心。

---

## §4.1 この章で切れる選択肢

ML パイプライン本体:
- **Vertex AI Pipelines**（KFP / TFX を実行するマネージド基盤）
- **Kubeflow Pipelines（KFP）**（SDK / コンポーネント定義）
- **TFX**（TensorFlow 中心の E2E パイプライン）

ワークフロー連携・スケジューリング:
- **Cloud Composer（Apache Airflow マネージド）**
- **Cloud Scheduler**（cron 起動）
- **Eventarc**（イベント駆動起動）
- **PipelineJobSchedule**（Vertex Pipelines 自身の定期実行）

CI/CD・再現性:
- **Cloud Build**（ビルド / テスト / デプロイの CI/CD）
- **Vertex ML Metadata / Model Registry**（成果物・系譜・モデル管理）

---

## §4.2 技術の最短定義

| 要素 | 一言定義 |
|---|---|
| Vertex AI Pipelines | KFP / TFX で定義した ML パイプラインをサーバーレスで実行するマネージド基盤。成果物を ML Metadata に自動記録。 |
| Kubeflow Pipelines (KFP) | パイプラインを Python SDK でコンポーネント化・DAG 化する仕組み。Vertex AI Pipelines 上で動かすのが定番。 |
| TFX | TensorFlow 前提の E2E ML パイプライン（ExampleGen〜Pusher）。データ検証・skew 検出を内蔵。 |
| Cloud Composer | Airflow のマネージド版。**複数システムをまたぐワークフロー**のオーケストレーションに使う。 |
| Cloud Scheduler | cron 形式で定期的にジョブ/HTTP を起動する。単純な定期起動向け。 |
| Eventarc | GCS 書き込みや Pub/Sub 等の**イベントを契機**にワークフローを起動する。 |
| Cloud Build | ソース変更を契機にビルド・テスト・デプロイを回す CI/CD。MLOps の CT/CD の足回り。 |
| Vertex ML Metadata | パイプラインの入出力・実行・系譜（lineage）を記録し、再現性・監査を支える。 |

---

## §4.3 試験で選ぶ条件

### ML パイプライン本体

| 選択 | 選ぶ条件 | 試験キーワード |
|---|---|---|
| Vertex AI Pipelines | マネージドに ML パイプラインを実行 / サーバーレス / 成果物自動追跡 | `serverless ML pipeline`, `Vertex Pipelines`, `pipeline components` |
| Kubeflow Pipelines | コンポーネント / DAG を SDK で定義（Vertex 上で実行） | `KFP`, `components`, `reusable pipeline` |
| TFX | TensorFlow 中心 / データ検証 / skew 検出を組み込みたい | `TFX`, `TensorFlow`, `ExampleGen / Pusher`, `data validation` |

### オーケストレーション・起動契機

| 選択 | 選ぶ条件 | 閾値 |
|---|---|---|
| Cloud Composer | **複数サービス/システムをまたぐ**ワークフロー、依存・retry・SLA・バックフィル | ML 単体でなく前後の ETL 等を含む全体統合か |
| PipelineJobSchedule / Scheduler | パイプラインを**定期実行**するだけ | 単純な cron でよいか |
| Eventarc | **イベント駆動**（新データ到着で起動） | トリガが時間でなくイベントか |
| Cloud Build | コード変更で**再学習/再デプロイ**を回す（CI/CD・CT） | パイプラインの変更契機がソース更新か |

---

## §4.4 試験で選ばない条件

- **Composer を選ばない**: ML パイプライン単体を定期実行したいだけなら、重い Airflow 環境は過剰。Vertex Pipelines + スケジュールで足りる。
- **Vertex Pipelines を選ばない**: 純粋に **ML 以外のシステム横断**の依存制御・バックフィルが主目的なら Composer。
- **Scheduler を選ばない**: 「新しいデータが届いたら」という**イベント駆動**は Scheduler（時間ベース）では表現できない。→ Eventarc。
- **手動運用を選ばない**: 「再現性」「監査」「系譜」が問われたら、ノートブック手動ではなくパイプライン化 + ML Metadata（思想④再現性）。

---

## §4.5 ひっかけパターン

1. **Composer と Vertex Pipelines の混同** — 「ML の学習〜デプロイを自動化」だけなら Vertex AI Pipelines。「ETL も含む複数システムの全体ワークフロー」なら Composer。**スコープの広さ**で切る。
2. **Scheduler vs Eventarc** — 「毎晩」は Scheduler、「新データ到着時」は Eventarc。**時間契機かイベント契機か**。
3. **二重起動** — Composer で本線を回しつつ Scheduler でも同じパイプラインを起動して二重実行になる選択肢。起動経路は一本化する。
4. **CI と CT の取り違え** — コード変更でパイプラインを再ビルド/再デプロイするのが Cloud Build（CI/CD）。新データで再学習するのは Pipelines + トリガ（CT）。両者は役割が違う。
5. **「再現性」を無視** — 成果物追跡が要件なのに ML Metadata/Registry を含まない選択肢。

## §4.6 判断フローチャート

**何でオーケストレーションするか**

```
オーケストレーション対象は ML パイプライン単体か、複数システム横断か？
 ├─ ML 単体（前処理→学習→評価→デプロイ） → Vertex AI Pipelines（KFP/TFX）
 │   ├─ TensorFlow 中心でデータ検証も組込みたい → TFX
 │   └─ 一般的なコンポーネント構成 → KFP on Vertex Pipelines
 └─ ETL 等を含む複数システムの全体ワークフロー → Cloud Composer（Airflow）
```

**いつ起動するか**

```
起動契機は？
 ├─ 定期（毎晩・毎週） → Cloud Scheduler / PipelineJobSchedule
 ├─ イベント（新データが GCS/Pub/Sub に到着） → Eventarc
 └─ ソースコードの変更（再学習/再デプロイ） → Cloud Build（CI/CD・CT）
※ 起動経路は一本化し、二重起動を避ける。
```

## §4.7 代表問題パターン

> 本書の自作シナリオ。

### 問題 1

> あるチームは「前処理 → 学習 → 評価 → モデル登録 → デプロイ」という ML ワークフローをマネージドに自動化し、各ステップの成果物と系譜を自動で記録したい。インフラ管理は最小にしたい。最適な方法は？
>
> A. Compute Engine VM 上で cron + シェルスクリプト
> B. **Vertex AI Pipelines（KFP）で各ステップをコンポーネント化し、ML Metadata で系譜を記録**
> C. Cloud Composer で全部を Airflow タスク化
> D. ノートブックを手動で順番に実行

- **正解: B**。決定打は「**ML ワークフロー単体**」「**マネージド**」「**成果物・系譜を自動記録**」（思想①④）。Vertex Pipelines は ML Metadata に自動記録する。
- A を切る: 自前運用で再現性・系譜が弱い。
- C を切る: ML 単体に Airflow は過剰（Composer は複数システム横断向け）。
- D を切る: 手動・再現性なし。

### 問題 2

> ML の再学習パイプラインは、**新しい学習データが GCS バケットに到着したとき**に自動で開始したい。固定時刻ではない。最適な起動方法は？
>
> A. Cloud Scheduler で毎時起動
> B. **Eventarc で GCS のオブジェクト作成イベントを契機にパイプラインを起動**
> C. 人が気づいたら手動実行
> D. Composer の daily DAG

- **正解: B**。決定打は「**データが到着したとき**」= イベント駆動。Eventarc が直接対応。
- A / D を切る: 時間ベースでは「到着時」を表現できず、無駄起動か遅延が出る。
- C を切る: 自動化要件に反する。

### 問題 3

> 複数の前処理ジョブ（Dataflow）、BigQuery 変換、外部 API 連携、そして ML 学習を、**依存関係・リトライ・SLA 付きで一つのワークフロー**として統合運用したい。既存チームは Airflow に慣れている。最適な方法は？
>
> A. Vertex AI Pipelines だけ
> B. **Cloud Composer（マネージド Airflow）で全体を DAG 化（ML 部分は Vertex Pipelines を呼び出す）**
> C. Cloud Functions を多数つなぐ
> D. Cloud Build のステップに全部入れる

- **正解: B**。決定打は「**複数システム横断**」「**依存・リトライ・SLA**」「**Airflow に慣れている**」。Composer の適合条件が揃う。
- A を切る: Vertex Pipelines は ML 中心で、ETL や外部連携の全体統合には Composer が適切。
- C を切る: 関数の手組みは依存・SLA 管理が脆弱。
- D を切る: Cloud Build は CI/CD であり常時のデータワークフロー運用基盤ではない。

---

## §4.8 最後に覚える一文

> **ML 単体の自動化は Vertex AI Pipelines（系譜は ML Metadata）、複数システム横断は Cloud Composer。定期起動は Scheduler、イベント駆動は Eventarc、コード変更の再デプロイは Cloud Build。起動経路は一本化する。**
