## C.1 まず思い出す: 7 つの設計思想（→ [序章 §0.2](00_試験の世界観.md)）

迷ったら、これに寄せた選択肢が正解になりやすい。

1. **マネージド優先** — 自前構築より Vertex AI / BigQuery / AutoML
2. **既存データ所在地を尊重** — BQ にあるなら BQ 内で完結を疑う
3. **本番運用前提** — monitoring / retraining / lineage まで考える
4. **再現性重視** — notebook 単発でなく pipeline 化・metadata
5. **責任ある AI** — fairness / bias / explainability / privacy を軽視しない
6. **スケール要件で分岐** — batch/online、CPU/GPU/TPU、endpoint/container
7. **組織横断利用** — Feature Store / Model Registry / Experiments

## C.4 頻出ひっかけ（混同しやすいペア）

| 混同しやすい | 切り分けの決定打 |
|---|---|
| skew ↔ drift | baseline が**学習データ**なら skew、**過去の本番データ**なら drift |
| RAG ↔ fine-tuning | **知識（最新/社内）**は RAG、**振る舞い（文体/形式）**は fine-tuning |
| Vertex Pipelines ↔ Composer | **ML 単体**は Pipelines、**複数システム横断**は Composer |
| Scheduler ↔ Eventarc | **時間契機**は Scheduler、**イベント契機**は Eventarc |
| Sampled Shapley ↔ Integrated Gradients | **微分不可能(ツリー/表形式)**は Shapley、**微分可能(NN)**は IG |
| CMEK ↔ VPC-SC | **鍵の自社管理**は CMEK、**持ち出し境界**は VPC-SC |
| SA 鍵 ↔ WIF | 外部/別クラウドから・鍵を配らないなら WIF |
| Online ↔ Batch | レイテンシ要件があれば Online、大量・非同期なら Batch |
| Monitoring ↔ XAI ↔ Evaluation | ズレ検知=Monitoring、理由=XAI、良し悪し=Evaluation |
| 蒸留 ↔ fine-tuning | コスト/レイテンシ削減=蒸留、振る舞い適応=fine-tuning |
